Architecture sketch · synthesizes the five-harness study + KNOWLEDGE-WORK-HARNESS.md

The skeleton from five coding harnesses. The commitments nobody has shipped.

An ideal knowledge-work agent harness inherits six primitives from pi, Claude Code, codex, code_puppy, and opencode — and adds four architectural commitments those harnesses never had to make. This page sketches the components and how they fit together, flags what is borrowed versus invented, and names the questions that more research will need to answer before any of this is built.

Borrowed · six primitives Loop, sub-agents, tools, permissions, persistence — table stakes lifted from the coding harnesses. Mature, well-understood, low-risk to clone.
Novel · four commitments Layered memory, proactive scheduler, provenance lineage, draft as first-class object — these are where the engineering investment lives, and where open questions remain.
01Thesis

Knowledge work breaks three coding-harness assumptions.

Verification is no longer free (no compiler, no test suite). Blast radius is no longer recoverable (a sent email is unsendable). Source-of-truth is no longer the filesystem (it's a scattered, conversational substrate). Each of these reshapes a layer of the harness. The architecture below is what falls out — six layers cloned, four layers invented.

Verification asymmetry

Pressure

A coding agent gets tsc, pytest, and the next tool call as ground-truth checks. A knowledge-work agent gets text back from a read_email call — no pass/fail. The loop has to model verification itself: critique sub-agents, draft-then-refine cycles, explicit checkpoints.

Blast-radius asymmetry

Pressure

rm -rf node_modules is annoying. A misrouted email or triple-booked CEO meeting is socially expensive and not undoable. Permissions can no longer treat all writes as equivalent — a small minority of actions need preview-then-confirm as the default.

Source-of-truth asymmetry

Pressure

Coding agents anchor to the filesystem and git. Knowledge-work facts live in a Slack thread, a draft Notion page, an unread email reply, and what was said an hour ago. Memory cannot be punted to "summarize when full" — the conversation is the substrate.

03Components

Six borrowed primitives, four novel commitments.

Each card is a component from the architecture map. Inherited cards cite the specific file and pattern in the source harness; novel cards describe a behaviour that doesn't exist in any of the five and flag the deepest open questions. None of the novel components are speculative — they all fall directly out of the verification, blast-radius, or source-of-truth pressures.

Orchestration Loop

Inherited
From Claude Code · src/query.ts queryLoop
+ pi · agent-loop.ts:170 outer/inner pattern

An AsyncGenerator that yields events as work happens, with pi's two-loop split: outer iterates follow-up turns, inner handles mid-turn steering. The yield boundary is the user's only window into a 30-minute deep-research operation that otherwise looks like a spinner.

  • Steering interrupts mid-turn without resetting context
  • Yields StreamEvent, Message, compaction-progress
  • Survives process restarts via JSONL replay

Sub-Agent Variants

Inherited
From opencode · agent/agent.ts AgentVariants
+ Claude Code · Task tool for isolation

Five variants share session history but get distinct system prompts and permission scopes. Variants are cheap (same model, different prompt). research spawned via Task-style isolation keeps fifty pages of scraped reading out of the main context.

  • primary, research, draft, critique, summarize
  • critique is read-only, looks for unsupported claims
  • Variant model could expand: schedule, negotiate, brief

Tool Registry

Inherited
From code_puppy · _builder.py two-pass
+ codex · turn.rs built_tools() per turn

MCP-first surface for SaaS-shaped knowledge work (Gmail, Notion, Linear, Drive, Calendar, Slack). Two-pass registration probes native tool names first, then loads MCP and filters collisions. Per-turn re-registration handles mid-session OAuth, scope, and workspace changes.

  • Native: save_draft, read_memory, spawn_variant
  • Name collision = silent shadow → catastrophic in knowledge work
  • Tool set is not stable across long sessions

Permission Gate

Inherited & reframed
From codex · exec_policy.rs GranularApprovalConfig
+ opencode · once/always/reject
+ Claude Code · plan mode as preview

Codex's two-axis shape, re-axed for blast radius rather than filesystem write. Four action classes with different friction defaults. Preview is part of the permission UX — for external sends the modal shows the actual email before any approval can be granted.

  • read / local-draft / external-send / destructive
  • once-per-thread is the humane default, not once-forever
  • preview-before-send is the new "plan mode"

Session Log

Inherited
From pi · ~/.pi/agent/sessions/{id}.jsonl
+ codex · reverse-replay reconstruction

Append-only JSONL is the substrate. Diffs well, forks well (a fork is the prefix copied to a new file), debuggable with jq. Resume scans backward to the most recent compaction checkpoint then re-applies selective forward events.

  • Branching via parentId — every fork preserved
  • Safe for concurrent writes
  • Compaction writes a checkpoint, doesn't truncate

Working Context

Inherited
From Claude Code · src/query.ts three-tier compaction
+ codex · AutoCompactWindow

The window the LLM actually sees this turn. Micro/auto/snip cascade keeps it under budget. Critically, compaction writes a checkpoint into the session log rather than dropping the trimmed messages — they're still recoverable from disk.

  • Visible compaction events (yielded to UI)
  • Re-hydrates from session log on resume
  • System context (git status analog) becomes "today's calendar"

Persistent Memory

Novel
No analog in the five · closest is CLAUDE.md (static, global)

A vector store + semantic index over every artifact, decision, draft, and turn-level snippet across sessions. Retrieval auto-fires at turn start, conditioned on the current question. Each chunk carries source, retrieval time, confidence, and a contradiction link if a later turn disagreed with it.

  • Staleness signal: 6-month-old "deadline" facts decay
  • Cross-session: "last Tuesday Bob said…" is recoverable
  • Provenance on every chunk (see Provenance Layer)

Proactive Scheduler

Novel
No analog in the five · closest is codex's submission_loop (event-driven, no scheduling primitive)

A cron/webhook/poll layer that wakes the agent on schedule or external signal: monitor a customer's tickets, summarize merged PRs every Monday, alert on competitor filings. Each scheduled job carries a budget cap and idempotency keys so a runaway watcher can't drain $400 overnight or re-alert on the same event.

  • Triggers: cron, webhook, RSS, polling, agent self-schedule
  • Budget enforcement: $/tokens/calls per job
  • Goal-state update — agent can refine its own watch criteria

Provenance Layer

Novel · cross-cutting
No analog in the five · academic LLM research has versions, none productionized as harness-level concern

Every retrieved fact carries (source_id, source_url, retrieved_at, confidence). The LLM emits inline citations the UI renders as clickable hover-cards showing the source snippet. Synthesis preserves lineage: "Q3 deadline is May 15 [src:gmail-thread-abc]". Citations must survive compaction — they re-attach to summarized chunks.

  • Source ledger as a join table on memory chunks
  • Lineage graph: synthesis preserves source DAG
  • Critique variant audits unsupported claims

Draft State Machine

Novel
No analog in the five · closest is opencode's tool-call state machine (operates on tool calls, not artifacts)

"Draft email, get approval, revise on edits, send" is one logical operation in three phases — not three message turns. Drafts are first-class versioned entities, separate from messages. The send action binds to a specific version; ratifying a different version requires re-confirmation. Inline edits, comments, ratings all route back to the draft variant as structured input.

  • States: pending → reviewed → sent, with version deltas
  • User feedback ≠ chat message — it's structured edit input
  • Send is bound to (draft_id, version)
04One request, traced

"Draft a board-update email to the investors, include Q3 highlights."

Following one realistic request through the architecture. Steps are color-coded to the component they touch. Open tags mark places where the design report flagged a question that still needs research.

01 orchestration loop

User submits the prompt. The outer loop accepts it as a follow-up; the inner loop opens a new turn.

The queryLoop yields a TurnStarted event to the UI. The user immediately sees activity: this matters because the next steps will take a while.

02 persistent memory

Turn-start retrieval fires against the persistent vector store, conditioned on the prompt.

Returns: prior board updates, investor preferences ("Linda likes bullet form"), Q3 financial decisions made in past sessions. Each chunk arrives with its provenance quadruple. open · what triggers retrieval?

03 sub-agent · research

Primary spawns a research variant in isolation to gather Q3 highlights from external sources.

Research variant uses MCP tools to pull from Notion (Q3 OKR doc), Linear (closed milestones), and Gmail (customer quotes). Reads about 80 pages of source material; only the synthesis returns to the main context.

04 tool registry · permission gate

Each notion.search, linear.list_issues, gmail.search call passes the permission gate as a read.

Auto-allowed, silent. No friction. Every result attaches a source_id as it enters memory.

05 provenance layer

The research synthesis emits inline citations; the lineage graph records "Q3 highlight #3 = synthesis(notion:abc, linear:xyz)".

The synthesis chunk now has both upstream sources and a derivation note. open · how granular?

06 sub-agent · draft

Primary delegates composition to the draft variant, passing the synthesis + investor preferences.

The draft variant produces v1 of the email. The draft state machine creates a Draft artifact, persisted to the artifact index. The body links to its source citations.

07 sub-agent · critique

Primary auto-spawns critique on draft v1 — read-only, looking for unsupported claims and tone issues.

Critique reports two unsupported claims and a tone mismatch. Primary either revises automatically (producing v2) or surfaces to user. open · how to choose?

08 draft state machine · user surface

Draft v2 surfaces in the user's draft editor. User makes inline edits; the editor emits structured edit events.

Edits route back to the draft variant which produces v3 as a delta. The draft is now in the reviewed state. open · concurrent edit conflicts?

09 permission gate · external send

User approves send. Permission gate triggers external-send path: modal with the full rendered v3 preview, recipient list, send button.

User confirms. A grant is issued bound to (draft_id, version=3). If the user later approves v4, that requires re-confirmation.

10 tool registry

gmail.send executes. Draft state machine transitions to sent (v3); artifact index updates.

Sent action is now an immutable artifact with its own provenance pointer back to the draft, the synthesis, and the original Q3 sources.

11 session log · artifact index

JSONL has captured every event. The artifact index has a Sent record. Memory ingests the new draft and outcome for next time.

Six months later, "what did I send the board about Q3?" hits the persistent memory and returns the sent artifact with its full citation tree intact.

05Open questions

What more research has to answer before this is buildable.

The four novel components each carry hard open questions. Borrowed components also leave decisions open where the source-harness pattern doesn't transfer cleanly. These aren't paper cuts — most of them gate the architecture. Listed by area, ranked by how load-bearing they are for v1.

Q01memory · tier 3

What goes into the vector store, and when?

Every turn? Only artifacts? Just user-ratified facts? Indexing everything risks polluting recall with noise; indexing only artifacts loses the implicit context ("we decided X in a side conversation"). Need: empirical study of recall quality vs. index granularity on real knowledge-work sessions.

Why it matters: Wrong choice means either drowning the agent in irrelevant past context, or losing the conversational substrate that justified building memory in the first place.
Q02memory · eviction

How do you decide what to forget — and is that decision reversible?

Vector stores grow. A staleness signal is the obvious starting point, but "stale" facts often become re-relevant. Hard-delete loses the audit trail; soft-delete bloats the index. Need: a policy model that distinguishes archive (still queryable, downweighted) from retract (user explicitly disowned).

Why it matters: Forgetting wrongly is worse than not remembering at all — a confidently-recalled stale fact is dangerous in a way a missing fact isn't.
Q03memory · trust

How does the agent calibrate source reliability?

A fact from a CEO email and a fact from a public web scrape are not equally trustworthy. Provenance metadata can carry a source-tier, but the calibration has to come from somewhere. Manual config? Learned from user corrections? Default trust hierarchy by source type? Need: a default model + a feedback loop.

Why it matters: Without calibration, the agent will treat "blog post says X" with the same weight as "the CFO said X in writing", which is a recipe for confident hallucination.
Q04scheduler · budget

What's the right unit and scope for budget caps?

Per-job? Per-day? Per-user? Tokens, dollars, tool calls, wall-clock? A watcher that polls every 5 minutes can be cheap-per-call but expensive-per-month. Need: a multi-dimensional budget model with hard ceilings and soft warnings, plus a clear UX for "your watcher hit its cap, do you want to extend?".

Why it matters: Without this, proactive loops are a fast path to surprise bills and user loss-of-trust.
Q05scheduler · idempotency

How does the agent know it has already acted on this signal?

A churn-risk watcher must not re-alert on the same ticket. The naive answer is a dedupe key per (signal, action) pair, but real signals shift slightly each poll. Need: a semantic-equivalence check, not a string match — and a model of when "this is the same thing, just updated" vs. "this is genuinely new".

Why it matters: Re-alerting trains users to ignore the agent. Missing genuine new signals defeats the point of the watcher.
Q06provenance · UI

How do you surface citations without making the UI unreadable?

Inline brackets work for academic readers; they're alienating for "draft me an email" users. Hover cards, footnotes, color-coding the spans — each has trade-offs. Need: design exploration with real users on whether to render citations inline, on demand, or only when contested.

Why it matters: If users don't see citations, the provenance layer has built infrastructure for nothing. If they're too loud, users disable them and we're back to ungrounded synthesis.
Q07provenance · synthesis

What level of granularity does lineage track? Sentence? Claim? Document?

"X is true because S₁ ∧ S₂" is elegant but breaks down when synthesis fuses six sources into one paragraph. Per-sentence lineage is expensive; per-paragraph loses precision. Need: a tradeoff study and probably a hybrid (paragraph-level by default, sentence-level on demand when a claim is challenged).

Why it matters: Lineage too coarse is non-actionable; lineage too fine is unaffordable to compute and store.
Q08drafts · concurrency

What happens when the user edits while the agent is revising?

Y-CRDT (Yjs) is the obvious primitive. But the merge model for "agent generated v3, user concurrently typed in v2" is non-trivial — operational transforms on AI-generated prose don't always make semantic sense. Need: a UX model for "agent revision in flight, you may want to wait" and a fallback for hard conflicts.

Why it matters: Without a model here, the draft state machine becomes "whichever side wrote last wins", which silently destroys work.
Q09permissions · friction

How much preview-and-confirm friction is too much?

"Always ask, always preview" is safe but tedious for a power user sending fifty emails. "Once-per-thread" softens it, but threads can drift (the email I approved was to Bob's group; now you want to bcc Eve). Need: scope-aware approvals — defined by recipient set, content category, send window — not just by tool name.

Why it matters: Friction in the wrong place trains users to mash "approve" by default, defeating the whole point of the gate.
Q10memory · multi-tenant

Is the persistent memory single-user, or shared across a team?

"What did we decide about the Acme deal?" is a great team query. But cross-user memory needs access control (per-source, per-recipient), conflict resolution (two PMs decided different things), and a model of who can write what. Need: a deliberate stance — single-user-only for v1, or design for multi-tenancy from day zero (the latter is much more invasive).

Why it matters: Retrofitting multi-tenancy onto a single-user memory is a rewrite. Designing for it adds substantial complexity that may be wrong for solo-knowledge-worker use cases.
Q11long-horizon · decomposition

How does the agent decompose a multi-day goal into a schedule of sub-tasks?

"Investigate competitor X for two weeks and report back" needs a plan, sub-goals, midway check-ins. None of the five harnesses model this. Plan mode is single-turn. Scheduler runs single tasks. Need: a goal-state object the scheduler can read and the agent can refine over time — closer to a project than a task.

Why it matters: Without this, "proactive" collapses to "scheduled" and the harness misses its highest-value use case.
Q12loop · observability

What's the right user-facing model of a 30-minute deep research?

Yielded events let us show something, but a stream of "fetching… fetching… fetching…" is no better than a spinner. The user needs a model of progress: how far through, what's left, what's been found, when to expect output. Need: a progress representation that's more than logs but less than a Gantt chart.

Why it matters: Long-horizon tasks fail on user patience, not on capability. The progress UX is load-bearing for the harness being usable at all.
