Q07 / 12 Provenance · Granularity

At what granularity should source lineage be tracked through synthesis?

Why it matters Per-sentence is precise but expensive and — counter-intuitively — degrades attribution quality. Per-paragraph is cheaper but coarse. Per-document loses the whole point. Get this wrong and you either pay 5× the storage for false precision or strip audit value entirely.
01Research findings

Surprising consensus: "finer is better" is wrong. Sentence-level enforcement degrades attribution by 16–276%.

The granularity question has become a major area of empirical research in 2024-2026, with surprising consensus that intuitions about "finer is better" are wrong. The Johns Hopkins paper Are Finer Citations Always Better? (Wang et al., 2025) found that enforcing sentence-level citations degraded attribution quality by 16-276% compared to the optimal setting, which was paragraph-level (k=4-8 sentences). Larger models suffer most: a 120B-parameter model improved 96.7% moving from sentence to paragraph granularity. The mechanism is straightforward: sentence boundaries disrupt coreference chains and multi-sentence inference, forcing the model to either over-cite (every sentence borrows from many sources) or fabricate plausible-looking single-source attributions. Task-dependent results matter too: factoid reasoning (HotpotQA) peaks at k=4 sentences, while summarization (GovReport) prefers k=16.

Anthropic's Citations API takes a sophisticated approach: it chunks documents at the sentence level on the input side, but lets Claude chain consecutive sentence spans on the output side to cover paragraph-sized claims. The clever part is that cited_text does not count against output tokens (only inline citation markers do), which means precision is essentially free on the output side once chunking is set up. Perplexity, by contrast, uses inline claim-level numbered citations attached during generation rather than post-hoc, but ZipTie/drainpipe analyses note its known failure mode is "misattribution": the synthesis is correct but the citation marker points at the wrong source, suggesting claim-level mapping is fragile when many sources are interleaved.

The faithfulness research is sobering. Correctness is not Faithfulness in RAG Attributions (Wallat et al., 2024) found up to 57% of "correct-looking" citations are post-rationalized: the model generated the answer from priors and then attached a plausible-looking citation afterward. Finer granularity makes post-rationalization worse, not better, because the model has to invent more citation hooks per token. Sub-sentence proposals try to fix this by demanding "concise and sufficient" spans, but require a separate extraction pass that adds latency and another point of failure.

For memory compaction specifically, the Provenance-Aware Tiered Memory work frames the core dilemma: "write-time compression forces irreversible retention decisions under uncertainty." Once you summarize, you cannot reconstitute sentence-level lineage. The proposed solution is a two-tier system where summarized memory carries paragraph-level pointers back to a verifiable cold store, with a runtime sufficiency check that pulls the original chunk when needed.

02Options — pick one

Sentence is too expensive; paragraph is too coarse alone. The winning patterns layer them.

Option 1

Sentence-default everywhere

Every claim in every artifact carries one or more sentence-level source IDs, propagated through all synthesis and compaction steps.

Each generated sentence gets a citation array stored alongside the text in the vector memory. Compaction must preserve per-sentence lineage; if a summary sentence merges three source sentences, all three IDs attach. Storage is roughly 3-5x the text size.

Pros
  • Maximum auditability; user can trace any claim to one or two source sentences
  • Matches what Perplexity users expect from "AI-grounded" output
  • No second pass needed when a claim is challenged
Cons
  • 16-276% attribution quality degradation per JHU paper; model over-cites or fabricates
  • 2-5x storage and 1.5-2x output token cost
  • Post-rationalization rate climbs above 50%; precision is illusory
  • Compaction either kills lineage or balloons summaries
Option 2

Paragraph-default with cached source spans

Lineage is tracked at the paragraph level for synthesis, but each paragraph caches its supporting source spans (the raw quoted text) so finer granularity can be reconstructed locally without re-querying.

Every drafted paragraph stores: (a) ordered list of source-document IDs it draws from, (b) the actual extracted spans (Anthropic-style cited_text) that were retrieved to write it. On compaction, paragraph-level lineage propagates intact; sentence-level reconstruction is a local Claude call against the cached spans. Storage is ~1.5-2x text size.

Pros
  • Matches empirically optimal model granularity (paragraph)
  • Cheap output-token cost; one citation set per paragraph
  • Survives compaction cleanly
  • Sentence-level on-demand stays local (cached spans), no live re-retrieval
Cons
  • "Which sentence?" answers are reconstructed, not authoritative
  • Cached spans add storage even when never queried
  • Post-rationalization still possible at paragraph level (just less measurable)
Option 3

Pure hybrid — paragraph default, sentence on challenge

Generation and storage are paragraph-level by default; if a user challenges a specific claim, an on-demand worker re-runs the relevant generation step with Anthropic's Citations API enabled to get sentence-level evidence.

Default writes carry one source-set per paragraph. When a user clicks "verify this" on a sentence, the harness reconstructs the input context for the paragraph (from the audit log), calls Claude with sentence chunking, and stores the resulting sentence-level lineage as an augmentation. Storage stays lean unless challenged.

Pros
  • Lowest steady-state storage and token cost
  • Reconstruction matches the actual model behavior at challenge time
  • Aligns with how humans actually audit (spot-check, not exhaustive)
Cons
  • Requires keeping enough audit log to re-run generation (defeats some compaction savings)
  • Re-generation may produce different sentence-level attributions than the original
  • Adds verification latency (seconds) at exactly the moment a user is suspicious
  • Faithfulness is reconstructed, not preserved — arguably worse than no audit
Option 4

Tiered provenance memory (TierMem-style)

Recommended

Two-tier memory: warm tier holds paragraph-level lineage with cached spans; cold tier preserves full sentence-level provenance and original documents indefinitely. A runtime sufficiency check escalates from warm to cold when a query needs precision.

Drafts and synthesized artifacts live in warm memory with paragraph lineage + span cache (per Option 2). Original retrievals, raw research notes, and the first-pass draft live in cold storage with sentence-level lineage and never get compacted. A retrieval-time check (claim confidence + user trust level + claim sensitivity) decides which tier to serve. Compaction operates only on warm tier.

Pros
  • Best fit for the four novel harness primitives (especially layered memory and draft-as-artifact)
  • Compaction is safe by construction (cold tier always intact)
  • Audit always returns authoritative answers (cold tier never reconstructed)
  • Cost scales with how much you draft, not how much you read
Cons
  • Two storage systems to maintain and keep consistent
  • Cold tier grows unboundedly; needs its own retention policy
  • Sufficiency check is itself a model call (small but nonzero)
Recommendation

Option 4 — Tiered provenance memory.

For a knowledge-work harness whose four novel primitives include both layered cross-session vector memory and draft-as-first-class artifact, the granularity decision cannot be made independently of compaction and the draft lifecycle. Option 1 fights the model and the storage budget; Option 3's "reconstruct on challenge" sounds elegant but is exactly the failure mode the faithfulness papers warn about (post-hoc rationalization dressed up as audit). Option 2 is the right default behavior, but it needs Option 4's cold tier behind it to make compaction safe and audits authoritative. The two tiers map cleanly onto your existing primitives: warm tier is the working memory the proactive scheduler reads, cold tier is the provenance substrate the lineage primitive guarantees.

The concrete shape: at write time, store paragraph-level citations with cached spans in warm, and append the full sentence-level chunk-to-claim mapping (plus original retrieval response) to an append-only cold log keyed by draft-version. At read/synthesis time, the scheduler works from warm only. On "verify this claim," the audit path skips reconstruction entirely and queries the cold log by (draft_id, paragraph_id, sentence_offset). This sidesteps the 57% post-rationalization rate because the cold record is the original generation, not a re-derivation, and it lets the warm tier stay cheap enough that compaction is genuinely useful instead of a lossy compromise.
