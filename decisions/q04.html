Q04 / 12 Scheduler · Budget

What budget primitives should proactive jobs expose — and how do they fail safely?

Why it matters Without principled caps, proactive loops are a fast path to surprise bills and user loss-of-trust. The cautionary tales are vivid — a $607 Replit bill from an unattended session; a multi-agent loop that scaled from $127/week to $47,000/week in eleven days. The right primitives prevent both runaway spend and silent stuck loops.
01Research findings

Providers moved away from hard caps. Agent operators have to layer their own — multi-dimensional, with pressure warnings.

The state of the art for LLM budget controls in 2025-2026 reveals a striking gap: most major providers have moved AWAY from hard spending caps toward soft alerts, leaving production agent operators to roll their own enforcement. OpenAI quietly changed monthly budget limits from hard cutoffs to alerts-only in 2025, leaving developers to layer proxies (LiteLLM, custom gateways) for actual ceilings. Anthropic exposes workspace-scoped spend limits but documents them as "spend notifications" rather than hard kills, with the one exception being the Claude Code workspace, which uniquely supports per-user monthly spend limits. Vercel AI Gateway provides team budgets ($200 default) with 50/75/100% webhook notifications, but explicitly notes that exceeding the limit does NOT automatically stop usage unless a separate pause-projects toggle is enabled.

The runaway-cost incidents driving 2025-2026 design thinking are vivid: a $607 Replit Agent bill from an unattended autonomous session; a multi-agent loop that scaled from $127/week to $47,000/week in eleven days because two agents recursively asked each other for clarification. These have crystallized a consensus: budgets must be MULTI-DIMENSIONAL (cost in dollars, tokens, wall-clock, step count, and tool calls) because each catches a different failure mode — token budgets prevent runaway prompts, step budgets prevent infinite loops, time budgets prevent stuck workflows, and cost budgets prevent cascading spend. Single-dimension caps consistently miss real-world failures.

Workflow runtimes (Inngest, Trigger.dev, Hatchet, Temporal) handle ADJACENT concerns well — concurrency keys for fair scheduling, throttling per-user via expression keys, step-level retries — but none of them natively enforce dollar/token budgets. Inngest's step.ai.infer adds telemetry but the cost guard still lives in your code. The most sophisticated open-source budget enforcement lives in LLM proxies: LiteLLM exposes max_budget, soft_budget, and budget_duration per key, per team, and per end-user, with overlapping windows ($10/day AND $100/month on the same key), hard-rejecting requests with ExceededBudget when exceeded.

The emerging pattern from production circuit-breaker designs is a three-layer gateway: a token-bucket throttler for volume, circuit breakers triggered by cost-per-request thresholds, and a fallback chain that gracefully degrades (smaller/cheaper model, disabled tools, cached responses, reduced reasoning depth) rather than failing hard. Cloud-cost UX precedent (AWS Budgets) standardizes graduated alert thresholds at 50/75/90/100% with both actual and forecasted signals, routed to high-signal channels (Slack, PagerDuty) rather than email alone, and supports auto-actions at hard thresholds.

02Options — pick one

Single-dimension caps miss the failures that matter. The recommended path is multi-dimensional + graceful.

Option 1

Single-Dimension Dollar Cap (Replit-style)

Each scheduled job declares a USD ceiling; harness kills the job when cost-to-date exceeds it.

When a watcher fires, the harness computes cost in real time from per-model pricing tables and aborts the job if cumulative spend crosses the cap. Caps stack at job, day, and workspace scopes — the tightest binding wins. User UI shows one number per scope.

Pros
  • Simple mental model for non-technical knowledge workers
  • Maps directly to bill-shock prevention (the actual user fear)
  • Easy to default and document ("$0.50/run, $5/day per workspace")
Cons
  • Misses infinite-loop and stuck-tool failures that haven't burned cash yet but will
  • Single number obscures runaway-token vs runaway-toolcall failure signatures
  • Real-time cost calc requires provider pricing tables that drift
Option 2

Multi-Dimensional Execution Budget (+ graceful degradation)

Recommended

Every job carries a typed budget tuple — {dollars, tokens, wall_clock_seconds, model_calls, tool_calls} — and the first dimension to exhaust ends the job, with model-downgrade thresholds before hard kills.

Job spec declares limits per dimension; harness tracks each independently and emits structured "budget pressure" warnings to the model when 75% of any dimension is spent (giving the agent a chance to wrap up cleanly). Scopes layer (per-job, per-user/day, per-workspace/month). On 75% workspace spend, auto-downgrade model class; on exhaustion, checkpoint state, post notification, and offer a one-click "extend budget and resume" to the user.

Pros
  • Catches all four documented failure modes (runaway prompt, infinite loop, stuck tool, cascading spend)
  • Pressure-warning UX lets the agent finish the task instead of failing mid-stream
  • Maps cleanly to provenance/draft artifacts — partial drafts survive budget exhaustion
  • Aligns with knowledge-work expectations (degraded synthesis > no synthesis)
Cons
  • More knobs means more decisions for the user (hide behind smart defaults)
  • Implementing pressure-injection requires harness-side prompt manipulation
  • Tracking five dimensions per job adds bookkeeping overhead
  • Model downgrade can silently degrade output quality without user awareness
Option 3

Tiered Budget with Graceful Degradation (3-layer gateway)

Soft caps trigger model downgrade or tool-pruning before hitting hard kill ceilings — same UX pattern as Option 2 but with dollars as the only dimension.

Three thresholds per scope. At 50% spend: notify-only. At 75%: degrade — switch from Opus to Sonnet (or Sonnet to Haiku), disable expensive tools, cap reasoning depth, prefer cached responses. At 100%: hard kill with checkpoint. Inspired by AWS Budgets actions plus circuit-breaker literature.

Pros
  • Preserves user experience during budget squeeze instead of failing outright
  • Aligns with cost-aware routing patterns already used by AI gateways
  • Simpler than Option 2 — one dimension to reason about
Cons
  • Model downgrade can silently degrade output quality without user awareness
  • Misses loop/stuck-tool failures that Option 1 also misses
  • Degradation paths must be tested per workflow type
Option 4

Proxy-Enforced Dollar+Token with Job-Local Step Caps

Borrow LiteLLM's per-key dollar+token model for spend, layered with harness-local step and wall-clock limits per job invocation.

Each user/workspace gets a virtual API key with max_budget (USD), soft_budget (alert), budget_duration (per day/month), and TPM/RPM limits. Per-scheduled-job specs additionally declare max_steps, max_wall_clock, and max_tool_calls enforced inside the harness. Spend control is provider-agnostic; flow control is harness-native.

Pros
  • Cleanly separates "money safety" (proxy) from "control flow safety" (harness)
  • Reuses battle-tested LiteLLM enforcement instead of reimplementing
  • Per-key model gives clean per-workspace/per-user isolation
Cons
  • Two enforcement layers means two places to debug when a job is killed
  • Adds a hard proxy dependency in the request path (latency, reliability)
  • Doesn't natively support degradation — kills are binary
Recommendation

Option 2 — Multi-Dimensional Execution Budget, extended with Option 3's graceful-degradation tier.

Knowledge-work scheduled jobs (watch inbox, monitor news on topic X, draft weekly digest) are uniquely vulnerable to all four failure modes: a draft generator can token-bloat, a research watcher can tool-call-loop on a slow source, a triage job can wall-clock-hang on a flaky webhook. A single-dollar cap (Option 1) reproduces the Replit failure mode where the cap fires AFTER the damage. A proxy-only model (Option 4) misses step/loop failures and conflates "money safety" with "the work was actually useful" — for knowledge work, a $0.30 watcher that loops 400 times and produces nothing is worse than a $0.30 watcher that produces a partial draft. The multi-dimensional budget plus pressure-warning gives the model the chance to checkpoint a draft artifact before exhaustion, which compounds with your "draft as first-class versioned artifact" primitive.

The UX surface should be opinionated: ship two preset tiers (Light: $0.10/run, 30s, 20 steps; Heavy: $1/run, 5min, 100 steps), expose a one-line override per watcher, enforce a workspace-level monthly hard ceiling that defaults to $20 (the bill-shock floor most consumer users tolerate), and emit 50/75/100% notifications. At 75% workspace spend, auto-downgrade to a cheaper model class and notify; at 100%, pause all watchers and require explicit reactivation. Make extension a single tap from the alert — the AWS Budgets lesson is that friction-free extension prevents users from disabling budgets entirely.
