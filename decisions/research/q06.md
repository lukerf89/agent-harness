# Q06 · Provenance · UI surfacing

## Research findings

Mainstream RAG products have converged on a hybrid pattern: **inline numbered superscripts** anchored to a **separate source panel/sidebar**, with **hover previews** filling the middle ground. Perplexity is the canonical example, using teal-colored superscripts ([1], [2]) inline with the answer text, source cards stacked in a side panel, and metadata (favicon, title, domain) for scanability. ChatGPT Search uses similar inline numbered links but at a much lower density (Perplexity averages ~22 citations per answer vs ChatGPT's ~8), suggesting different beliefs about how much sourcing readers actually want. Glean Deep Research and Notion Q&A both pair line-by-line citations with a panel; Granola 2.0 added inline citations with "jump to source" links pointing back into the meeting transcript, and uses hover-magnifier popovers for paraphrased/quoted context.

The Anthropic Citations API takes a different stance: it returns structured citation objects (with start/end character offsets and `cited_text`) and explicitly leaves *presentation* to the developer. This is significant — Anthropic, Glean, and Granola all treat citations as **structured data attached to spans**, not as visible glyphs in the text. That separation is what enables the same response to render as superscripts in a chat UI, as footnotes in a PDF export, or as nothing at all when pasted into Gmail.

Academic UX research is genuinely double-edged. The Stanford/UPenn study *Citations and Trust in LLM Generated Responses* (AAAI 2025) found that **the mere presence of citations raised trust significantly — even when the citations were random and irrelevant** — but trust dropped sharply once users actually clicked through and inspected them. The 2025 Brown/MIT "Cited but Not Verified" benchmark found frontier models maintain ~94% link validity but only 39–77% factual accuracy, and that accuracy *degrades* as citation count scales (~42% drop from 2 to 150 retrievals). The implication: **citation density is a vanity metric**; what matters is whether the user can cheaply verify when they choose to. ShapeofAI's catalog of citation patterns explicitly warns that inline-heavy designs risk "premature convergence on a single source" and recommends matching specificity to context.

A constraint specific to knowledge-work harnesses is **graceful degradation on export**. Drafts get pasted into Gmail, Slack, Word, or Notion — surfaces that don't render hover cards, don't preserve footnote anchors, and often strip markdown. Users actively use tools like "AI Text Cleaner" and "Clean Paste" to remove markdown noise and invisible characters from LLM output before sending. Any citation UI that *only* lives inside the harness will be either lost or visually corrupted the moment the artifact leaves it.

## Options

### Option 1: Perplexity-style inline superscripts + persistent source panel
**Summary:** Numbered superscripts ([1], [2]) at end of each claim, with a source panel showing favicon/title/snippet, mirroring Perplexity.
**How it works:** Every factual span is annotated server-side with one or more citation IDs. The renderer inserts subtle colored superscripts after the punctuation, and a right-hand panel lists numbered sources with favicon, title, domain, and a 1-line snippet. Hovering a superscript highlights its source card.
**Pros:**
- Familiar pattern — users coming from Perplexity, You.com, ChatGPT Search recognize it immediately
- High verifiability: every claim has a direct anchor
- Sidebar gives sources persistent presence without polluting the text
**Cons:**
- Visually noisy for long-form drafts (briefs, emails) where 20+ superscripts ruin readability
- Strips badly when pasted: `[1]`, `[2]` become orphaned numbers with no panel
- Research shows density does not equal accuracy and can create false-confidence

### Option 2: Color-coded spans with on-demand hover cards (no visible glyphs)
**Summary:** Sourced claims get a subtle underline or background tint; hovering reveals a citation card with quote + source. No numbers, no superscripts in the body text.
**How it works:** Each claim is wrapped in a span with a citation ID. Rendering uses a 1px dotted underline in a muted accent. On hover (or tap on mobile), a popover shows the source title, favicon, quoted passage, and a "go to source" link. Granola's hover-magnifier and Sana's popovers are precedents.
**Pros:**
- Clean reading experience — drafts look like prose, not Wikipedia
- Pasted output degrades to plain prose (the underline disappears, no orphan numbers)
- Users who don't care about provenance can ignore it entirely; users who do can hover any spot
**Cons:**
- No persistent at-a-glance source list — harder to skim "where did this come from overall"
- Hover doesn't work for screen readers, keyboard-only, or print/PDF without extra affordances
- Lower visible "trust signal" — research shows visible citations boost trust even when fake

### Option 3: Hybrid — invisible-by-default spans with a togglable density slider
**Summary:** Same colored-span model as Option 2 plus a global "Show citations: off / subtle / inline / footnote" control, with a persistent panel always available.
**How it works:** Citations are always present as structured metadata on spans. The user (or output format) picks the rendering: "off" for clean email drafts, "subtle" underline for in-app reading, "inline" superscripts for research briefs, "footnote" for exported reports. The panel is always one click away regardless of mode. Anthropic's API model — structured citation data, presentation-agnostic — is the architectural precedent.
**Pros:**
- Matches the artifact type: emails get clean prose, research briefs get academic-style refs
- Export formats can pick the right rendering (footnotes for .docx, panel for in-app, none for clipboard)
- Handles the strongest finding from UX research: visible citations build trust, invisible ones support verification — letting the user toggle gets both
**Cons:**
- More to build: needs a rendering layer, an export pipeline, and a settings affordance
- Risk of decision fatigue if exposed as a per-document choice rather than a smart default
- Discoverability — users may never find the toggle

### Option 4: Contested-only citations with always-on panel
**Summary:** No citations in the prose by default; the model surfaces citations only for claims it flags as low-confidence, contested, or the user explicitly challenges.
**How it works:** Each claim has a confidence score. High-confidence claims render clean; low-confidence ones get a small warning marker. Users can right-click any sentence to ask "source?" and get a hover card. The full source panel is always present.
**Pros:**
- Maximally clean output, ideal for drafts meant to be sent
- Frames citations as a verification *tool* rather than visual proof
- Aligns with research that "more citations" doesn't equal "more truth"
**Cons:**
- Loses the documented trust boost from visible citations
- Requires reliable per-claim confidence scoring (which is itself unreliable)
- Users in regulated domains (legal, finance, medical) expect visible provenance everywhere

## Recommendation
**Pick:** Option 3 — Hybrid (invisible-by-default spans with a density slider)

For a knowledge-work harness whose outputs span emails, briefs, research reports, and meeting summaries, no single citation density is correct — and the research strongly suggests that defaulting to "Perplexity-everywhere" creates citation noise without commensurate trust. The architectural insight from Anthropic's Citations API is the right one: **store citations as structured data on spans, decide presentation per artifact**. Default to subtle underlined spans inside the harness (Option 2's clean reading experience), but ship two non-negotiable extras: (a) an always-available source panel for users who want to skim provenance, and (b) export rendering rules — clipboard strips to plain prose, Word/PDF exports add footnotes, in-app share preserves spans. This directly solves the "drafts get pasted out" problem that breaks superscript-based designs.

The novel "source-provenance lineage on every claim" primitive should be implemented at the *data* layer (every claim span has a citation list with confidence, source URL, quoted text, retrieval timestamp) and the UI is just one consumer. This also positions the harness for the "contested-only" mode (Option 4) as a *user preference*, not a separate design — power users in clean-draft mode can opt into "show citations only when I challenge" with the same underlying data model.
