# Q03 · Memory · Source trust calibration

## Research findings

How modern systems calibrate source trust splits along a spectrum from purely manual (hand-tuned weights per source type) to fully learned (online updates from user behavior), with most production systems sitting in a hybrid middle. Glean's Enterprise Graph is the most mature reference: it uses click signals, dwell time, anchor scores, data-source affinity, role/department, and an organizational people-graph to rank — but critically, all of this layers on top of strict per-source ACLs rather than a single global "trust score." Perplexity's RAG pipeline assembles confidence from roughly four pillars (trustworthiness, authority, corroboration, provenance) and one analysis decomposed its weighting as ~40% authority + 35% freshness + 25% schema/structure. Notably, the field has shifted from domain-based authority (PageRank-style) to author-based authority — domain-authority correlation with citation likelihood dropped from r=0.43 (pre-2024) to r=0.18, while author-attributed content gets 58% more citations than anonymous content (Metrics Rule, 2025).

Academic 2024-2026 work confirms LLMs are bad at this by default. AuthorityBench (arxiv 2603.25092) shows models frequently fail to discount low-authority sources and treat unreliable content equally to authoritative material; the most effective interventions were explicit authority annotations in prompts and retrieval-layer authority ranking, not relying on the LLM to infer trust. KG-RAG calibration research (arxiv 2601.09241, "When to Trust") found naive RAG exhibits overconfidence despite factual errors and proposed causality-aware calibration that adjusts confidence based on provenance edges. Memory-architecture work (MemOS, Mem0) recommends attaching structured metadata — source, timestamp, sensitivity, author, version — to every memory unit so retrieval can be metadata-gated rather than purely semantic.

Google's E-E-A-T is the most-studied authority framework, and Google itself is explicit that there is no single E-E-A-T "score." Instead, dozens of signals (author bylines, citations to the author elsewhere, content corroboration, structured data) get blended; AI Overview citations are 2.3x more likely to pick high-E-E-A-T pages. The relevant lesson for a knowledge-work harness: even the most sophisticated system in the world avoids a single scalar trust number per source and instead keeps multiple orthogonal signals available at ranking time.

Temporal weighting is treated as orthogonal to authority. Standard practice in agent memory (Mem0, Letta-style systems) is exponential decay with a tunable half-life (e.g., 0.995/hour), applied as a soft rerank multiplier between roughly 0.3x and 1.5x — recency does not override authority, it modulates it. The challenge is that "more recent = more trustworthy" is wrong for many knowledge-work artifacts (a 2-year-old signed contract beats a Slack message from yesterday), which argues against pure recency-bias.

## Options

### Option 1: Static source-type weights
**Summary:** Operator-configured trust score per source type (email = 0.9, internal doc = 0.8, public web = 0.5), applied as a multiplier at rerank.
**How it works:** A YAML/JSON config maps each integration to a trust prior. At retrieval, the final score is `semantic_similarity * source_trust * recency_decay`. The user can override per-domain (e.g., trust nytimes.com > random blog). No learning loop. Source-type is the only authority signal beyond content match.
**Pros:**
- Trivially explainable and debuggable — a user can ask "why did you trust this?" and see the static weight
- Zero cold-start problem; works from day one
- No risk of feedback poisoning or drift from a few outlier corrections
**Cons:**
- Ignores that "email" can mean spam or the CEO; treats all instances of a source type identically
- Operator burden to tune and maintain; weights become stale as the user's workflow shifts
- Misses the strongest signal in knowledge work — *who* said it, not *where* it lived

### Option 2: Pure learned trust from corrections
**Summary:** Treat every user correction ("that was wrong," "trust this more") as a training signal; learn per-source-instance trust online.
**How it works:** Each memory or source carries a learned weight initialized neutrally. When the user corrects a generated answer, the harness assigns negative reward to the sources cited and positive reward to whatever the user provided instead, using a simple bandit or RLHF-lite reward model (à la TeachMe / Memory-assisted Prompt Editing). Over time the system learns that one specific Substack is more reliable than another even within the same "blog" source type.
**Pros:**
- Adapts to the individual user's actual reality (their CEO, their consultants, their newsletter sources)
- Captures fine-grained source-instance trust without manual config
- Improves over time without operator work
**Cons:**
- Cold-start: useless for the first weeks, when the user most needs to build confidence in the system
- Vulnerable to poisoning from a few misclicks or biased corrections (the "poisoned confidence calibration" risk flagged in the long-term-memory survey)
- Hard to explain to the user why a source is now distrusted

### Option 3: Authority graph from org metadata
**Summary:** Derive trust from a knowledge-graph of people, roles, and document provenance — Glean-style — rather than from the source channel.
**How it works:** Build an entity graph where every fact links to its author (resolved across email, doc, calendar). Author nodes carry attributes (role, seniority, recency-of-collaboration, frequency-of-correctness). Trust for a recalled fact = function(author_authority_in_topic, document_recency, corroboration_count_from_other_sources). A statement from the CEO inherits her authority node; a public blog inherits low authority unless the author is identifiable and elsewhere-cited.
**Pros:**
- Captures the real signal in a workplace — *who* is the authority on what
- Naturally supports corroboration (multiple independent authors saying the same thing boosts confidence)
- Aligns with where industry is heading: Perplexity author-based authority, Glean people-pillar, E-E-A-T 2025
**Cons:**
- Requires nontrivial entity resolution across systems (email-author = Slack-user = doc-author)
- Cold-start until the graph fills out; small/early users get little benefit
- Risks encoding org politics ("CEO is always right") and entrenching bias

### Option 4: Layered hybrid — typed prior + author graph + corroboration + learned override
**Summary:** Combine all three: a static source-type prior bootstraps trust on day one, an authority graph refines it per author, corroboration boosts agreed-upon facts, and learned overrides from user corrections adjust persistent outliers. Recency is a separate orthogonal modulator.
**How it works:** Each fact in vector memory stores `{source_type, author_id, timestamp, ingestion_provenance}`. At retrieval, score = `relevance × (w1·type_prior + w2·author_authority + w3·corroboration_boost) × recency_modulator × (1 + learned_delta)`. The learned_delta is bounded (e.g., +/-0.3) so a handful of corrections cannot flip a high-authority source to untrusted. Type prior fades in weight as the author graph fills out and as learned signal accumulates (Bayesian-style: more data = less reliance on prior). Recency is multiplicative, not additive, and never dominates authority for "stable fact" classes (contracts, decisions, policies) — only for "current state" classes (today's calendar, latest email thread).
**Pros:**
- Works from day one (type prior) but improves with use (graph + learned)
- Matches the empirical pattern from production systems (Glean, Perplexity) which all use multi-signal blends
- Resilient: no single signal can be exploited or corrupted; corroboration provides a sanity check
- Supports the harness's *provenance lineage* primitive — every score is decomposable into its contributors, so the user (or operator) can audit "why was this trusted?"
**Cons:**
- Most complex to build and tune; many hyperparameters
- Risk of opacity unless score decomposition is exposed in the UI
- Needs entity resolution and a corroboration index, both nontrivial infrastructure

## Recommendation

**Pick:** Option 4 — Layered hybrid (typed prior + author graph + corroboration + bounded learned override)

For a knowledge-work harness specifically — where one of your novel primitives is already *source-provenance lineage on every claim* — Option 4 is the only choice that compounds with the rest of the architecture instead of fighting it. You're already paying the cost to track provenance per fact; the marginal cost of also storing `author_id` and computing corroboration is small, and the payoff is large. Option 1 is too crude for the email-vs-blog distinction the user asked about (it can't tell CEO-email from intern-email). Option 2 alone is dangerous in a knowledge-work setting because the first weeks of use are when the user is hardest-pressed to catch errors — exactly when a learned-only system has no signal. Option 3 is closest in spirit but brittle on cold start.

The decisive design choice is to make trust a *vector*, not a *scalar*. Store the contributing signals (source-type prior, author-authority, corroboration count, recency, learned delta) separately, blend them at retrieval, and surface the decomposition when generating citations. This lets the harness say "I trusted this because: (a) CEO is the author, (b) corroborated by 3 other docs from the strategy folder, (c) written 2 weeks ago" — which is the kind of explainability that distinguishes a tool a knowledge worker will actually rely on from one they have to second-guess. Bound the learned override so feedback poisoning is contained, and let the type-prior weight decay as the author graph and learning signals mature (a Bayesian-shrinkage pattern). This also aligns with where the industry frontier is converging in 2025-2026: away from single scalar trust toward orthogonal, decomposable signals — Glean, Perplexity, AuthorityBench, and Google E-E-A-T all point the same direction.
