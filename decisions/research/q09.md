# Q09 · Permissions · Grant scope

## Research findings

Permission UX on consumer platforms has converged on a small set of temporal scopes after a decade of iteration. iOS started with binary allow/deny in iOS 5, added "While Using" in iOS 8, added "Allow Once" in iOS 13, and added "approximate vs precise" location toggles in iOS 14. The throughline is decomposing one big "always" grant into smaller dimensions the user actually cares about: time window (once / while-using / always), fidelity (precise / approximate), and surface (notification / background). macOS Sequoia took this further for screen recording, prompting weekly and on every reboot — which generated significant user backlash and was ultimately walked back, illustrating the cost when granularity tips into nagging without payoff.

OAuth scope design shows the same maturation curve. Slack explicitly migrated from "umbrella bot scope" to granular `object:action:perspective` scopes (e.g., `chat:write:user`) because the umbrella model caused install drop-off. GitHub's fine-grained PATs added a *resource* dimension on top of permission verbs — a token can be limited to one repo, with read or read/write on specific permission categories. Plaid layered another dimension: account-level scoping selected in the OAuth flow itself, so the user picks "checking yes, credit card no" at consent time. The shared lesson: scopes are multidimensional (verb × resource × principal × time), but if you expose all axes at once, users grant everything to escape the dialog — Plaid's research and broader OAuth literature both emphasize this rubber-stamping failure mode.

For agents specifically, the 2025–2026 industry has converged on tiered risk-based gating rather than per-tool prompting. OpenAI Operator/ChatGPT Agent explicitly asks before "actions of consequence" (purchases, sends, financial transactions) but proceeds silently for reads, achieving 99.9% confirmation recall on high-stakes sends and 100% on financial transactions per their system card. Anthropic Claude Code added "Auto Mode" in March 2026 specifically because users were approving 93% of permission prompts — fatigue had effectively defeated the prompt as a safety control. Auto Mode replaces regex allowlists with a classifier that reads *prose rules* and reasons about each action; it reports 0.4% false-positive rate on internal traffic. The MindStudio 5-rung ladder (Read → Suggest → Draft → Act-with-confirmation → Autonomous) and four-tier risk model (read-only / reversible-internal / external-recipient / irreversible-high-stakes) are the most-cited frameworks; both emphasize *reversibility* and *external recipients* as the key tier boundaries.

The consistent failure mode is "approve fatigue" — when users click through 90%+ of prompts, the prompt is theatre, not a control. The empirical mitigations are: (1) collapse low-risk actions into silent execution with audit log, (2) make the *content of the confirmation* meaningful (Superhuman's success isn't a modal — it's the 10-second undo plus a full draft preview before send; Boomerang and Gmail's undo-send use the same pattern), (3) scope grants along an axis the user already has a mental model for ("this conversation only," "Bob's team only"), and (4) let grants expire — the "Allow Once" iOS option became the most-used location grant precisely because it costs nothing to give.

## Options

### Option 1: Risk-tiered with silent low-risk
**Summary:** Classify every high-blast action into 4 tiers (read / reversible-internal / external-recipient / irreversible-financial) and only prompt for tiers 3–4.
**How it works:** The harness maintains a per-action-type risk classification configured in policy. Reads and reversible internal artifacts (drafts, scheduled-but-not-sent emails, internal notes) run silently with audit log. External-recipient actions trigger a content-aware confirmation showing the full draft, recipient, and time-of-send. Irreversible actions (sending money, deleting external data) require typed confirmation. No scope dimensions are exposed to the user — the classification is invisible until a tier-3+ action surfaces.
**Pros:**
- Mirrors the OpenAI Operator and Anthropic Auto Mode patterns that already work in production
- Zero cognitive load until an action genuinely matters
- Easy to explain ("we ask before anything reaches a real person")
**Cons:**
- One axis (risk tier) collapses recipient, content, and time into a single dimension — loses the ability to say "always OK to email Bob"
- Tier-3 prompts still arrive at every email send, which becomes fatiguing for users with high send volume
- Classification errors are invisible to the user until something goes wrong

### Option 2: Multidimensional scoped grants (the "Plaid model")
**Summary:** Expose a small set of scope dimensions — recipient set, content category, time window, thread — and let the user combine them when granting.
**How it works:** When the agent first attempts a high-blast action, show a structured grant dialog: "Allow [send email] to [these people] about [these topics] for [this duration]." Pre-fill with the narrowest sensible scope (this thread, this recipient, this hour) and surface "broaden" affordances ("also OK for Bob's whole team," "also OK for next 24h"). Grants are stored as policy and matched on subsequent actions; only out-of-scope actions re-prompt. Mirrors how Plaid lets users pick accounts, how GitHub fine-grained PATs combine repo × permission, and how iOS combines location × precision × duration.
**Pros:**
- Captures the dimensions that genuinely matter for knowledge work (a contract to a client and a status update to a teammate are not the same risk)
- Each grant amortizes across many future actions, dramatically reducing prompt volume
- Maps cleanly onto the harness's source-provenance lineage — scopes can reference the same recipient/topic entities
**Cons:**
- Complex grant UI risks the rubber-stamp problem OAuth literature warns about; users may "broaden to everything" to escape
- Requires the harness to reliably classify content category (status update vs contract) — misclassification escapes the grant
- More upfront design and policy storage complexity

### Option 3: Draft-as-default with explicit-send confirmation
**Summary:** The agent never sends — it always produces a draft artifact; sending is a separate user action that takes one tap with a Superhuman-style 10-second undo.
**How it works:** Leverages the harness's "draft as first-class versioned artifact" primitive. The agent autonomously composes, revises, and queues drafts; nothing reaches an external recipient without the user opening the draft and pressing send. For high-volume cases (replies under N words, internal Slack to known recipients) the user can configure auto-send-after-N-seconds with undo, mirroring Gmail's undo-send window. Scope decisions happen at *configuration time* per channel ("auto-send after 30s for replies to my team in #general"), not per action.
**Pros:**
- Eliminates the per-action prompt entirely; the user reviews drafts on their own schedule
- Aligns naturally with the harness's existing draft primitive — no new permission concept needed
- Matches the most-loved real-world pattern (Superhuman undo, Gmail undo-send)
**Cons:**
- Defeats much of the "proactive scheduler" value — agent can't actually close loops without user attention
- Bulk drafts can create their own form of fatigue (inbox of 40 drafts to review)
- Doesn't handle non-message actions well (calendar invites, payments) where "draft" is awkward

### Option 4: Risk-tiered + scoped grants for repeats (hybrid)
**Summary:** Default to risk-tiered prompts (Option 1), but every confirmation dialog includes an inline "also allow this for [recipient/thread/topic] [duration]" affordance that builds up scoped grants over time.
**How it works:** First time the agent emails Bob, the user sees a full preview and confirms. The dialog has a one-click "always OK for emails to Bob about this project for the next 30 days" toggle. Subsequent in-scope sends run silently with an unobtrusive notification; out-of-scope sends re-prompt. Mirrors iOS's "Allow Once / While Using / Always" — start narrow, let the user broaden in-context rather than upfront. Grants auto-expire and surface in a "permissions inbox" the user can audit and revoke.
**Pros:**
- Starts with proven risk-tier model; user only pays the scope-grant cost when they actually want to reduce future friction
- In-context grants avoid the upfront "approve everything" rubber-stamp failure
- Time-bounded grants prevent "permission rot" where stale broad grants outlive their justification
**Cons:**
- More implementation complexity than Option 1
- Risk that the inline "always allow" toggle becomes the new rubber-stamp button
- Scope evaluation logic needs to be fast and explainable, or users won't trust the silent path

## Recommendation
**Pick:** Option 4 — Risk-tiered + scoped grants for repeats (hybrid)

For a knowledge-work harness specifically, the per-action workload is high enough that pure risk-tiered prompting (Option 1) will hit Claude Code's 93%-approval-rate fatigue wall within weeks. Inbox triage, scheduling, and correspondence generate dozens of external-recipient actions per day per user — the very kind of volume that turned macOS Sequoia's weekly screen-recording prompt into a meme. At the same time, exposing a full multidimensional grant dialog upfront (Option 2) violates the most consistent finding in OAuth research: users grant everything to escape complexity. Option 3 cedes too much of the agent's value proposition; the whole point of a proactive scheduler with budget caps is that the agent closes loops without the user babysitting drafts.

The hybrid approach earns its complexity because it inverts the cost curve: the user pays a meaningful one-time confirmation (full preview, Superhuman-style undo window) and gets the option — but never the obligation — to convert that decision into a scoped grant they can audit later. Critically, grants must be (a) time-bounded by default (24h, 7d, 30d picker — never "forever"), (b) keyed to entities the harness already tracks via its provenance lineage (this recipient, this thread, this topic-tag), and (c) visible in a permissions inbox the agent surfaces weekly. Pair this with the "draft as first-class artifact" primitive: even auto-approved sends should briefly stage as drafts with an undo window, so the silent path still has a recovery hatch. This is the synthesis of what actually works at iOS, Plaid, Superhuman, and Operator scale — risk-aware defaults, in-context scope widening, and reversibility as the last line of defense.
