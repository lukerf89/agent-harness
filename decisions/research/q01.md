# Q01 · Memory · What gets indexed

## Research findings

The 2024-2026 literature converges on a clear taxonomy of memory write strategies, but production systems disagree sharply on the answer. MemGPT (Packer et al. 2023) treats memory as a hierarchical OS — the agent calls explicit `archival_memory_insert` / `core_memory_append` functions, indexing whatever the LLM judges worth promoting from main context to disk. MemoryBank adds an Ebbinghaus-curve decay so the system can forget and reinforce by time-elapsed and significance. Stanford's Generative Agents (Park et al. 2023) write *every* observation to a memory stream but score each with a 1-10 importance number from an LLM call; reflection fires only when accumulated importance crosses a threshold (~2-3x/day). A-Mem (Feb 2025) goes further still — every interaction triggers a "Zettelkasten note" with LLM-generated keywords, tags, contextual description, embedding, *and* links to related memories that get rewritten as new notes arrive. Mem0 (Apr 2025), now the de-facto production reference at 55k+ stars, runs a single-pass ADD-only extraction after each exchange, ranks on a tri-signal (semantic + BM25 + entity-link), and is explicit that "for most production applications, a hybrid approach works best: automatic retrieval at request start, agent-driven storage at turn end."

Shipping consumer products are notably more conservative than the papers. ChatGPT's memory has two distinct tiers: explicit "saved memories" (write triggered by user or by the model's judgment of likely future utility — name, role, ongoing projects, recurring preferences) and "chat-history references" (no extraction; just retrieval over raw transcripts since April 2025). Claude's Projects / Claude Code memory deliberately rejects vector indexing in favor of a transparent, file-based `MEMORY.md` directory; the source-code leak revealed a planned background "Chyros daemon" for continuous indexing but the shipped product writes synchronously during the session, biographical-and-procedural only, with files the user can edit. Notion AI's "memory" is even more explicit-only: it indexes the pages and databases the user designates plus an instructions page, and AI Meeting Notes treats every meeting as a first-class object. Glean indexes everything (content + people + activity) but that's enterprise search, not agent memory.

The strongest practical synthesis comes from Nicolas Bustamante's 2026 survey of Claude Code, Codex, and Hermes: synchronous live writes work for interactive agents with clean turn boundaries (Claude Code), while deferred two-phase batch extraction (Codex, after ~6h idle) wins for cloud-worker agents. Crucially, both authors and the LangMem docs converge on the same gate: *make empty output the default* — Codex's Phase 1 prompt explicitly rejects "one-off user queries with no durable insight, generic status updates, temporary facts that should be re-queried." This is the single discipline that keeps the store from drowning in noise. LangMem encodes the same split natively: a hot-path `create_manage_memory_tool` for agent-driven writes and a separate background consolidator.

For a knowledge-work harness specifically, three asymmetries matter. First, knowledge work has *natural high-value artifacts* coding agents lack: a sent email, a finalized draft, a meeting decision, a calendar accept — these are the closest analog to a committed git diff. Second, drafts are already a first-class versioned primitive in your design, so the memory tier doesn't need to re-derive what's "important." Third, provenance lineage on every claim means raw turns must be reachable anyway (for citation), so the persistent vector tier should index *interpretations*, not transcripts.

## Options

### Option 1: Ratified-artifacts only, written synchronously
**Summary:** Only commit user-touched artifacts (drafts saved, emails sent, decisions accepted, meeting summaries approved) to the vector tier, at the moment of ratification.
**How it works:** Every turn flows through working memory. When the user accepts a draft, signs off a decision, hits "send," or marks a meeting summary as final, a write-hook serializes the artifact plus its provenance graph into the vector store. No background extraction, no LLM importance scoring, no per-turn writes. Raw transcripts stay in cheap object storage indexed by session ID for citation lookup.
**Pros:**
- Zero noise — every entry has a human-confirmed ground truth
- Cheap (no per-turn LLM extraction call)
- Trivially aligns with provenance: each artifact already carries its lineage
- User-controllable by design (forgetting = un-ratify)
**Cons:**
- Loses tacit signals (user's recurring tone, preferred sources, complaints about a vendor) that never become artifacts
- Cold start for new users who haven't ratified much yet
- Misses agent-derived insights (synthesis claims, inferred preferences)

### Option 2: Hybrid with importance-scored extraction at yield boundaries
**Summary:** A Generative-Agents-style importance score (1-10) runs at every yield boundary; writes only fire above threshold, then a separate ratified-artifact write commits user-confirmed outputs at higher weight.
**How it works:** At each yield boundary (turn end where control returns to the user), a small/cheap model scores any new claims, preferences, or facts on importance and saves only those above threshold T into the vector tier with a `derived` flag. When the user ratifies an artifact, that artifact + provenance is written separately with a `ratified` flag and higher retrieval weight. Background consolidator (nightly) re-scores, dedupes, and demotes stale entries via MemoryBank-style decay.
**Pros:**
- Captures tacit signals and synthesized claims, not just artifacts
- Two-tier weighting lets retrieval prefer ratified content while still surfacing patterns
- Aligns with Mem0's empirically-best hybrid (auto retrieve, agent-driven write)
- Survives messy users who rarely ratify
**Cons:**
- Importance scoring is itself unreliable — LLMs score inconsistently
- Per-yield LLM call adds latency and token cost on every turn
- Two flags complicate retrieval ranking and eviction
- Risk of "memory pollution" if threshold drifts

### Option 3: Explicit save_memory tool, agent-driven, post-yield
**Summary:** Give the agent a `save_memory(content, type, source_refs)` tool; the agent decides what to retain, mirroring Claude Code, LangMem hot-path, and MemGPT.
**How it works:** No automatic indexing. The agent has explicit tools — `save_memory`, `update_memory`, `forget_memory`, `search_memory` — that it invokes during reasoning when something seems worth keeping. Typed entries (preference, fact, decision, project-state, contact) enforce schema. User sees writes in the trace and can veto.
**Pros:**
- Maximum transparency — every write is in the trace, auditable, user-vetoable
- Matches Claude Code's shipped pattern and LangMem hot-path quickstart
- No background pipeline needed; trivial to reason about
- Agent learns what to save from few-shot examples in its prompt
**Cons:**
- Quality depends entirely on prompt discipline — agents are notoriously bad at "default to no-op"
- Adds tokens/latency to every reasoning step that uses the tool
- Skips signals the agent doesn't think to record
- Doesn't naturally privilege ratified artifacts

### Option 4: Two-phase deferred extraction (Codex-style)
**Summary:** No writes during the session; an offline pipeline extracts memories from the full transcript + ratified artifacts at session idle (>N minutes) using a small model, then a larger model consolidates.
**How it works:** Sessions buffer raw transcripts. After an idle window, Phase 1 (cheap model) scans the transcript with a strict "will a future agent act better because of this?" gate, defaulting to empty output. Phase 2 (larger model) consolidates Phase 1 candidates against existing memory, dedupes, links, and writes to the vector tier. Ratified artifacts skip Phase 1 and go straight to consolidation.
**Pros:**
- No in-session latency cost; cheapest hot path
- Better extraction quality (full transcript context, slower model allowed)
- Hardest empty-output gate filters noise most effectively
- Plays well with your proactive-scheduler primitive (extraction runs in the same scheduled batch)
**Cons:**
- Memory not fresh — won't help within the same session
- Needs a clean session-end boundary; long-running knowledge-work sessions blur that
- Two-phase pipeline is operationally heavier
- Lag between user action and memory availability complicates user mental model

## Recommendation
**Pick:** Option 2 — Hybrid with importance-scored extraction at yield boundaries, with two adjustments: (a) treat ratified artifacts as a *separate, automatic* high-weight write path (Option 1 baked in), and (b) run a nightly background consolidation pass (the cheap part of Option 4) for dedup, link-building, and decay.

The knowledge-work context tilts the balance toward a hybrid because your other novel primitives already do half the work. Source-provenance lineage means raw transcripts must be addressable anyway, so the vector tier should index *condensed interpretations* — exactly what importance-scored extraction produces. Drafts as first-class versioned artifacts give you a free, high-precision write signal that Option 1 alone would over-rely on but that Option 2 elegantly slots in as the "ratified" tier. A pure agent-tool approach (Option 3) is tempting for transparency but every survey from Nicolas Bustamante to LangMem to Mem0 reports the same failure mode: agents over-save when given the tool, polluting the store within days. Pure deferred extraction (Option 4) loses the within-session adaptation that makes a *knowledge*-work agent feel intelligent across a single long research session.

Concretely: at every yield boundary run a cheap importance scorer on new claims/preferences with a high threshold (calibrated against an eval set, defaulting to "no-op" per the Codex discipline); on every ratification event write the artifact + provenance graph immediately with a higher retrieval weight; run nightly consolidation that dedupes, re-links A-Mem-style, and applies MemoryBank decay. Expose a `forget_memory` tool to the user (not the agent) for control. This mirrors Mem0's production-validated pattern while honoring the structural advantages of your knowledge-work primitives.
