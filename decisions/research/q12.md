# Q12 · Loop · Long-task observability

## Research findings

Across 2024-2026, every leading long-running agent product has converged on the same insight: **a chronological event stream alone destroys patience, but a stream paired with a visible plan sustains it for tens of minutes**. Perplexity's engineering team stated this most directly — when they instrumented Pro Search, users tolerated multi-minute waits *only* when the underlying plan was visible and expandable; this finding drove their entire "planning separated from execution" architecture (LangChain breakout-agent case study). Jakob Nielsen frames the same observation as response-time tiers: under 10s supports turn-taking, 1-10 min "feels conversational if narrated," and 10+ min "breaks the traditional interaction paradigm" and requires an operator-dashboard metaphor rather than a progress bar.

The dominant pattern in shipped products is **hybrid plan + live event stream with a focused/current node**. Devin's session UI splits into a Progress tab (the plan/steps with a current-step highlight) plus Shell/Browser/Editor tabs (the underlying event streams), so the user picks their altitude. Manus surfaces a persistent "Manus's Computer" panel (raw event stream) alongside a `todo.md` file that the agent edits live to check off plan items — making the plan a *first-class artifact* rather than just a UI widget. Cursor's Plan Mode and Claude Code's TodoWrite are functionally identical: a markdown checklist that the agent rewrites, streamed to the UI, with status icons (in-progress, done) so users see the cursor moving through the plan. Perplexity Deep Research shows a three-phase sidebar (interpret → execute → synthesize) with expandable steps; ChatGPT Deep Research as of Feb 2026 added real-time progress tracking with the ability to interrupt and refine.

A separate, equally important finding: **interim findings (cards as they arrive) outperform pure progress indicators for *research* specifically.** Perplexity surfaces sub-queries and intermediate sources as they accumulate; ChatGPT Deep Research streams a running source list; Manus emits artifacts (files, screenshots) into the side panel as it produces them. This serves two purposes — it (a) gives the user something to skim while waiting, reducing perceived latency, and (b) lets them course-correct mid-run if they see the agent going wrong. Nielsen calls these "conceptual breadcrumbs" — *synthesized insights at key milestones*, not just technical step logs.

What loses patience is well-documented: opaque "thinking…" displays (OpenAI's original o1 hidden chain-of-thought generated user backlash, which is why o3-mini was updated to expose summarized reasoning); generic spinners on multi-minute tasks ("a loading spinner on a 45-second AI task is an abandonment trigger" — Particula); and percent-done bars without an ETA or scope ("5% progress that would take half an hour is too long to sit and stare at" — Nielsen). The METR study on long agent tasks found measurable performance degradation past ~35 minutes of work — meaning the UX must also support *graceful interruption and salvage*, not just patient waiting.

## Options

### Option 1: Plan tree + live event stream (hybrid, two-pane)
**Summary:** Persistent left/top pane shows the research plan as a checkable tree with current-focus highlight; right/bottom pane streams events (tool calls, fetches, URLs) chronologically.
**How it works:** The agent writes its plan as a structured artifact at the start (sub-goals → sub-sub-goals), updates step status (pending/in-progress/done/skipped) as work proceeds, and emits raw events into a collapsible log beneath. The current node is visually anchored. This is Devin + Manus + Cursor + Claude Code, all of whom independently converged here.
**Pros:**
- Lets users zoom out (plan) or in (events) at will — handles both casual checkers and debuggers
- Plan-as-artifact doubles as the agent's working memory, supporting pause/resume and rollback
- The current-focus highlight serves as the "where are we" anchor that pure logs lack
**Cons:**
- Plans for true research are often *discovered*, not pre-stated — early plan is misleading and re-planning is jarring
- Two-pane UI is dense; can feel overwhelming on mobile or for non-technical users
- Risks teaching users to micromanage steps instead of trusting outcomes

### Option 2: Interim-findings cards (artifacts-first)
**Summary:** The primary view is a growing list of cards — each card a finding, source, sub-question answered, or draft fragment — with progress metadata (current phase, ETA, source count) as ambient secondary info.
**How it works:** Inspired by Perplexity Deep Research's source list and ChatGPT's evidence stream. As the agent works, every meaningful artifact (a source read, a claim extracted, a sub-question answered, a draft paragraph) emits a card. Users scroll/skim the accumulating output during the wait; the final report is composed from these cards. Plan and event log are available but secondary.
**Pros:**
- Inverts perceived latency — the user is *reading* the output rather than waiting for it
- Aligns perfectly with the harness's source-provenance lineage primitive (each card carries its evidence)
- Natural fit for the draft-as-versioned-artifact primitive — early cards become draft sections
**Cons:**
- Cards before synthesis can mislead: a half-formed finding looks authoritative
- Hard to convey "I'm 60% done" — no obvious progress anchor
- Requires the agent to emit well-formed artifacts continuously, raising agent-side complexity

### Option 3: Three-layer operator dashboard (Nielsen-style)
**Summary:** ETA + budget bar (top), critical-path node + plan tree (middle), recent events + interim findings (bottom) — all simultaneously visible, like a build/deploy dashboard.
**How it works:** Directly applies Nielsen's three-layer model. Top strip always shows: time elapsed, ETA window, budget consumed vs. cap, and a "stop and salvage" button. Middle pane shows the plan tree with the critical-path (gating) node highlighted. Bottom pane interleaves events and interim findings. Modeled on Vercel deploy dashboards and GitHub Actions live logs.
**Pros:**
- Honest about uncertainty (ETA windows, not false precision) and cost (budget bar pairs with the proactive-scheduler primitive)
- Salvage button + budget cap = principled handling of the 35-min degradation cliff
- Familiar metaphor for anyone who's watched a CI build — low learning curve
**Cons:**
- Dashboard density is high for knowledge workers who aren't engineers — non-coders may feel they're "watching a server"
- Critical-path concept assumes a DAG; pure research is more exploratory and the gating node may be unclear
- Top-strip ETA is hard to estimate honestly for open-ended research

### Option 4: Adaptive progressive disclosure (collapsed by default)
**Summary:** Single-line status pill ("Researching… 14 sources, ~3 min remaining") that expands on click to reveal plan, then expands further to reveal event stream.
**How it works:** Default view is one line plus a "show details" affordance. Mid-tier expansion shows the plan checklist (Cursor/Claude Code style). Full expansion shows event stream and interim findings. Background processing kicks in automatically at 60s (Particula's threshold), with notification on completion.
**Pros:**
- Respects users who want to fire-and-forget — patience problem partially solved by getting them out of the watching loop
- Scales cleanly from 30s to 30min tasks — same UI handles both
- Aligns with knowledge-work users' expectations (closer to Notion/email than dev tools)
**Cons:**
- Hidden-by-default can feel opaque; some users will distrust silent agents
- Loses Perplexity's documented insight that *visible* plans drive patience — collapsing the plan defeats the purpose
- Background-on-completion notification requires reliable cross-session delivery (email, push)

## Recommendation
**Pick:** Option 2 — Interim-findings cards, with a collapsible plan tree borrowed from Option 1 for users who want it

For a knowledge-work harness, the patience problem is solved most durably by *making the wait productive reading time*. Two of your four novel primitives (source-provenance lineage and draft-as-versioned-artifact) are already card-shaped — every claim has provenance, every draft is a versioned object — so the UI should mirror the data model. As the agent reads sources, surface them as cards with their provenance attached; as it forms claims, surface them as draft fragments that the user can already start editing or accepting. This converts the 10-30 minute wait from "watch a spinner" into "skim the working notes," which is what Perplexity discovered the hard way and what ChatGPT Deep Research's Feb 2026 redesign doubled down on. Layer in the plan tree as a collapsible secondary view (Devin/Manus/Claude Code pattern) so power users can zoom out, and adopt Nielsen's top-strip honesty about ETA and budget — but the *primary* surface should be the growing artifact stream, not the plan or the event log.

The reason to reject the pure dashboard (Option 3) is audience fit: non-coding knowledge workers will not warm to a CI-build aesthetic, and the harness's whole point is that knowledge work is the surface area. Option 1 (pure plan+log) treats the user as a supervisor; Option 2 treats them as a reader, which matches how researchers, writers, and analysts actually want to engage with a 20-minute task — they want to *see what it found*, not *what it did*. Pair Option 2 with a hard 60s threshold for offering background mode (Particula) and a visible budget cap that ties into the proactive-scheduler primitive, and you get a UI that is simultaneously honest about cost, productive during waits, and patient-supporting up to the ~35min degradation cliff METR identified.
