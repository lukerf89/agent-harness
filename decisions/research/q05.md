# Q05 · Scheduler · Idempotency

## Research findings

Workflow systems across the board converge on a remarkably uniform pattern: **the producer-side opaque idempotency key as the primary dedup mechanism**. Stripe assigns each event a unique `evt_...` ID that consumers store in a database with a unique constraint; GitHub stamps every webhook with `X-GitHub-Delivery`; Inngest uses a sender-provided event `id` with a 24-hour TTL; Trigger.dev exposes `idempotencyKeys.create()` with a configurable TTL defaulting to 30 days; Restate ties calls to a `WithIdempotencyKey` option. The shared principle is that the *source* (or the agent triggering the work) emits a stable identifier and the *sink* maintains a deduplication ledger keyed on it. Hookdeck explicitly warns that "deduplication doesn't replace idempotency in your application logic" — it's a load-reduction tool, not a correctness guarantee, because the underlying action must still be safely re-runnable.

Polling triggers — closer to the knowledge-work watcher case — work differently. Zapier requires the API to return items in reverse-chronological order with a unique `id` field, and maintains a cache of seen IDs. The crucial trick for *updated* items (e.g., a Linear issue gets a new comment): the integration "preserves the original `id` value before setting `id` to a new combined value that is unique for every update" — typically `originalId + updatedAt`. This means Zapier doesn't really do "state diffing"; it forces the source to fingerprint the version. n8n's Remove Duplicates node provides a more local mechanism with three modes ("within current input", "across previous executions", "clear history"), bounded by a default 10,000-item history per node, with field-level "All Fields" / "All Fields Except" comparison — essentially a content-hash strategy with explicit field selection.

Temporal takes a different angle: signal deduplication is *application-level*, because the workflow runtime only auto-dedupes retried client RPCs via request IDs. The recommended pattern is "treat signals like events in a distributed system: dedupe them" using a history set in workflow state and a custom idempotency key in the signal payload. Activities use the (WorkflowRunID, ActivityID) pair as a stable retry key. This shows the pattern most relevant to agent watchers: **the dedup ledger lives inside durable workflow state**, not in a separate cache, and the key is constructed from semantically meaningful inputs.

Recent (2025–2026) LLM-agent research on cycle/loop detection — including the arxiv paper *Unsupervised Cycle Detection in Agentic Applications* (2511.10650) — explicitly addresses the "have I seen this state" question. The most successful approach is **hybrid**: cheap structural fingerprinting of action/tool-call sequences first (F1 0.08 alone), augmented by semantic-similarity analysis over embeddings of state content (F1 0.28 alone), with the combination reaching F1 0.72. The structural layer catches exact repetitions cheaply; the semantic layer catches functionally-equivalent revisits with surface variation. Semantic-cache literature converges on cosine-similarity thresholds around 0.80–0.90 with explicit fallbacks when high similarity is misleading (e.g., long conversations producing spurious overlap).

## Options

### Option 1: Source-fingerprint composite key (Zapier model)
**Summary:** Build a deterministic composite key `(signal_type, entity_id, version_token)` per signal and store it in a bounded ledger keyed by `(watcher_id, key)`.
**How it works:** For each watched source, define an extractor that produces a stable entity ID (Linear issue ID, Gmail thread ID, ticket ID) plus a `version_token` derived from the source's own change marker — `updatedAt`, `last_comment_id`, ETag, revision number, or `state_changed_at`. The watcher computes `hash(entity_id || version_token || action_intent)` and consults a per-watcher LRU/TTL ledger before firing the action. The action is also wrapped in an idempotency key downstream (Inngest/Trigger.dev style) so any retry is a no-op.
**Pros:**
- Cheap, O(1) lookup; no embeddings or LLM calls per poll
- Matches how every production webhook/poller works — well-understood failure modes
- Naturally handles "reopened ticket" by including `state` or `status_changed_at` in the version token
- Forces the integration author to think about what "materially new" means for each source
**Cons:**
- Requires per-source adapter code (one extractor per Linear/Gmail/HubSpot/etc.)
- Pure-string version tokens miss "trivial" updates that the agent *shouldn't* act on (a typo-fix comment)
- Storage grows with entity count; needs explicit eviction policy
- No notion of "this is a slightly different signal but the same situation"

### Option 2: Entity-state snapshot diffing
**Summary:** Persist the last-acted-on snapshot per entity and on each poll compare structured fields against it; act only when designated fields change.
**How it works:** When the watcher acts on entity X, it stores a snapshot `{status, assignee, last_comment_id, due_date, sentiment_tag}`. On next poll, the new payload is normalized into the same shape and diffed. The watcher fires only if a *triggering* field changed (per-watcher rules: "act when status flips to closed", "act when a new external comment arrives"). This is essentially CDC for external APIs, application-level.
**Pros:**
- Most expressive — you can encode "act if Y changed but not Z"
- Naturally explains decisions ("I acted because status went open→closed at 14:02")
- Handles the reopened-ticket case directly via state transitions
- The snapshot doubles as provenance lineage (one of your novel components)
**Cons:**
- Heavy: requires a normalized schema per signal type, plus storage of full snapshots
- Triggering rules can be brittle as upstream APIs evolve fields
- Schema drift breaks dedup silently; needs version tags on snapshots
- Doesn't handle free-text content changes (new comment body) without an additional sub-strategy

### Option 3: Hybrid structural + semantic layered dedup
**Summary:** Two-tier check — fast composite-key/structural dedup first, then embedding-based "is this materially new" check for the cases the structural layer is uncertain about.
**How it works:** Layer 1 is Option 1's composite key — if seen, drop. Layer 2 runs when the key is unseen *but* the entity has been acted on recently (e.g., within N hours): embed the relevant new content (new comment text, email body, ticket summary) and compare cosine similarity against embeddings of prior signals for the same entity. Below a similarity threshold (~0.85), treat as materially new; above, suppress. Optionally a Layer 3 LLM judge for high-stakes ambiguous cases. This mirrors the arxiv hybrid cycle-detection result.
**Pros:**
- Catches "two comments saying the same thing in different words" that hash dedup misses
- Cheap path stays fast; expensive path only fires when needed
- Aligns with the harness's existing vector memory primitive — reuse infrastructure
- Tunable per-source thresholds; degrades gracefully
**Cons:**
- More moving parts; harder to debug a suppressed signal
- Threshold tuning is per-source and drifts over time
- Embedding costs and latency on every uncertain poll
- Semantic similarity is not the same as semantic equivalence — false positives can silently drop genuine new work

### Option 4: Event-sourced log with versioned watcher cursors
**Summary:** Treat every fetched payload as an immutable event, append to a per-watcher log, and let each action be a deterministic projection over the log with a checkpointed cursor.
**How it works:** The poller writes every raw response (or every new entity-version) into an append-only event store keyed by `(watcher_id, sequence_number)`. The agent action loop replays from a cursor; actions are themselves recorded as events with the input event sequence as their idempotency key. Snapshots collapse history periodically. This is the Temporal/event-sourcing pattern adapted for external polling.
**Pros:**
- Full audit trail for free — directly supports your provenance-lineage primitive
- Replayable: you can re-run the agent against historical signals safely
- Naturally handles versioned snapshots (one of your novel components)
- Strongest correctness guarantees — exactly-once becomes provable
**Cons:**
- Highest infrastructure cost (event store, snapshotting, schema versioning)
- Overkill for low-volume watchers (a personal inbox triage agent)
- Requires action handlers to be pure-deterministic projections, which constrains LLM use
- Operational complexity (snapshot evolution, schema migration) is significant

## Recommendation
**Pick:** Option 3 — Hybrid structural + semantic layered dedup

For a knowledge-work agent harness, pure source-fingerprinting (Option 1) is necessary but insufficient: real knowledge-work signals routinely differ syntactically while being substantively equivalent ("Any update?" / "Just checking in!" / "Bumping this"), and any production agent that pings a customer twice for the same nudge will burn user trust within a week. Pure embedding-based dedup (the temptation when you already have a vector memory layer) is too lossy on its own — semantic similarity routinely overrates near-duplicate boilerplate as "the same" while missing genuinely new information buried in a long thread. The 2025 cycle-detection research validates that the *combination* of cheap structural fingerprinting and selective semantic comparison materially outperforms either alone, and the harness already has the vector memory primitive needed for layer 2.

Concretely: implement Option 1 as the always-on first layer — a `dedup_ledger` table keyed on `(watcher_id, entity_id, version_token, action_intent_hash)` with a configurable TTL per watcher (24h–30d, mirroring Inngest/Trigger.dev). When Layer 1 says "unseen but same entity acted on recently," invoke Layer 2: embed the new content and check cosine similarity against the prior-action embedding for that entity; suppress above 0.88, escalate to a cheap LLM judge between 0.75–0.88, fire below. Wire every action through a downstream idempotency key so retries are safe regardless. This stays cheap on the 95% path, handles the genuinely hard cases the rigid-key approach silently fails, and the dedup ledger itself becomes part of the provenance lineage your harness already wants to maintain. Defer Option 4's full event-sourced log to high-stakes watchers (financial, legal) where the audit guarantees justify the operational tax.
