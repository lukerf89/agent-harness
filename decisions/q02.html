Q02 / 12 Memory · Eviction

How should memory be evicted, downweighted, or archived from a long-lived store?

Why it matters Forgetting wrongly is worse than not remembering at all — a confidently-recalled stale fact is dangerous in a way a missing fact isn't. But vector stores grow, and an index that's never pruned drifts into a swamp of contradictions and noise that degrades retrieval quality on every query.
01Research findings

The field has moved away from hard delete. Contradictions live alongside each other; retrieval decides.

The 2024-2026 literature has converged on a clear consensus: production-grade agent memory must "learn to forget" or it accumulates contradictions, hallucinated context, and retrieval noise. MemoryBank (Zhong et al., AAAI 2024) was the canonical early move — it applied an exponential Ebbinghaus-curve decay to each memory's "intensity" score, with reinforcement when the entry is accessed, summarized, or rehearsed. This established three signals every newer system reuses: time decay, access-recency (LRU/LFU), and LLM-judged semantic importance. A-Mem (Xu et al., 2025) extended this by making memories Zettelkasten-style notes that are continuously linked, re-tagged, and rewritten when new entries arrive — eviction is implicit because entries get absorbed into or superseded by their neighbours.

The contradiction problem is now the dominant theme. Mem0's most recent algorithm explicitly retreated from aggressive UPDATE/DELETE because "the more you update, revise, and compress, the more you risk distorting what actually happened." Its 2026 release uses ADD-only extraction: contradictions become parallel timestamped facts and retrieval logic decides which is currently valid. Zep/Graphiti goes further with a temporal knowledge graph where every edge has valid_from/valid_to and "invalidation" is a soft event (the fact remains, audit-visible). The STALE benchmark (2026) showed that even top models fail at "implicit conflict" — recognising that a new observation invalidates a stored belief without an explicit negation — which argues against trusting LLM-judged hard deletes for anything load-bearing.

MemGPT/Letta provides the architectural pattern that most production systems now copy: a small core memory (always in context), a recall memory (mid-tier searchable), and an archival memory (cold, searchable on demand). Movement between tiers is the eviction mechanism — "forgotten" data is rarely deleted, just demoted to a tier that's not auto-injected. ChatGPT's consumer UX shows the user-facing side: explicit memories are individually deletable from a settings panel and can be removed conversationally, with OpenAI retaining a 30-day shadow log of deleted memories for safety. Notion's enterprise approach is purely TTL-based: page deletion triggers embedding deletion within 60 days, with a 30-day soft-delete restore window.

For audit trails specifically, the database community's wisdom transfers cleanly: soft delete with tombstone timestamps is the right default for anything a user might reference, regulator might audit, or agent might need to re-justify a past claim with. Hard delete is reserved for explicit user retraction (GDPR-style) and for genuinely ephemeral entries. The re-relevance problem — stale facts becoming relevant again — is best handled by not deleting at all but by demoting (Zep) or downweighting via decay multipliers (Milvus Decay Ranker, LangChain's TimeWeightedVectorStoreRetriever).

02Options — pick one

Three viable architectures, from cheapest math to fullest audit.

Option 1

Pure decay-curve downweight (MemoryBank style)

Every memory has an "intensity" score that decays exponentially with time and gets reinforced on access; retrieval multiplies similarity by intensity — nothing deleted, but stale entries sink below the threshold.

Each entry stores created_at, last_accessed_at, and access_count. A scheduled job calculates intensity = exp(-Δt/τ) × (1 + log(access_count)) where τ is per-entry-type. The retrieval ranker is cosine_sim × intensity. Entries below an intensity floor get demoted to a cold tier but are not hard-deleted. Contradiction triggers a downweight of the older entry, not deletion.

Pros
  • Perfect audit trail — nothing silently lost
  • Re-relevance is automatic: stale topics resurface via semantic similarity
  • Mathematically simple, no LLM in the eviction loop, low latency
  • Aligns with MemoryBank/A-Mem research lineage
Cons
  • Index keeps growing forever; storage and ANN cost climb monotonically
  • Doesn't handle hard contradictions cleanly — both facts surface with similar scores
  • No GDPR-style "really gone" guarantee unless layered with explicit retraction
  • Tuning per-entry τ requires heuristics or calibration
Option 2

MemGPT-style tiered demotion with periodic compaction

Recommended

Three tiers (core, recall, archive); a background consolidator moves entries between tiers, marks contradictions as "superseded," and compacts the archive into clusters with a frozen audit blob.

New facts land in recall. A nightly job runs three passes: (a) demote unused recall entries to archive after N days without access; (b) detect contradictions via LLM judge over high-similarity pairs and mark older sides "superseded by X" without deleting; (c) compact archive clusters with >5 similar entries into one merged summary, storing originals in a provenance_blob column that's not vector-indexed but is fully audit-queryable. User explicit retraction is a hard delete from the index but leaves a tombstone row in the audit log.

Pros
  • Bounded index size — archive compaction prevents unbounded growth
  • Clear architectural mapping (core / recall / archive)
  • Provenance lineage and audit trail preserved in non-indexed cold storage
  • Contradictions handled with explicit "superseded" link, not silent overwrite
Cons
  • Consolidation is expensive (LLM calls per cluster); needs careful budget caps
  • Compaction can distort original phrasing — the risk Mem0 retreated from
  • Tier-promotion logic is fiddly; re-relevant archive entries need an "uplift" path
  • More moving parts to test and debug
Option 3

Append-only temporal graph (Zep/Graphiti style)

Memory is a bitemporal knowledge graph where every fact has valid_from/valid_to (world time) and recorded_at/invalidated_at (system time); nothing is ever deleted, but queries default to "valid as of now."

Each new observation creates an edge with valid_from=now. When the LLM detects a contradicting observation, it doesn't delete the old edge — it sets valid_to=now on the previous edge and adds a new one. Retrieval defaults to valid_to IS NULL but can be queried historically. User explicit retraction sets a redacted=true flag that masks content but preserves structural record.

Pros
  • Strongest audit story — full historical replay is trivial
  • Re-relevance is a non-issue: invalidated facts still findable, just filtered out by default
  • Handles contradictions natively and explicitly
  • Bitemporal model maps cleanly to source-provenance lineage
Cons
  • Graph operations are heavier than vector lookups; latency tax on every retrieval
  • Requires sophisticated extraction (entity + relation + temporal scope) — more LLM cost on write
  • Unbounded growth even worse than Option 1
  • Significant engineering investment; less common in productivity-tool stack
Recommendation

Option 2 — Tiered demotion with periodic compaction.

For a knowledge-work harness specifically — where the user needs to trust drafts, audit claim provenance, and where the agent is proactive (so memory writes happen without user-in-loop curation) — the tiered approach is the right balance of three of your four novel components. The recall→archive demotion gives bounded retrieval costs that respect the scheduler's budget caps. The "superseded by X" link rather than deletion gives the contradiction-aware provenance lineage. And the non-indexed provenance_blob for compacted entries is exactly the audit substrate you need for "why did the agent draft this claim three months ago?" forensics. Hard delete is reserved only for user explicit retraction, satisfying GDPR-style requirements without polluting the everyday path.

The reason to reject Option 1 (pure decay) is that knowledge work is not chat — users will hold the agent accountable for a draft it produced six months ago, and "the source memory decayed below threshold" is not an acceptable explanation. The reason to defer Option 3 (full bitemporal graph) is that it's the architecturally purest answer but represents a 2-3x engineering investment compared to Option 2; once Option 2 is in production and you have observability on which contradictions actually matter, you can selectively upgrade specific entry types (preferences, project facts) to bitemporal edges without rewriting the whole memory layer. Treat Option 3 as a v2 evolution, not a v1 commitment.
