Q11 / 12 Long-horizon · Decomposition

How should the agent decompose a multi-day goal into sub-tasks?

Why it matters Without this, "proactive" collapses to "scheduled" and the harness misses its highest-value use case. The cautionary tales are documented: AutoGPT and BabyAGI showed how naive recursion produces infinite loops; the goal-drift literature shows every frontier model deteriorates over long horizons; production agents like Devin had to invent a "living plan" to recover.
01Research findings

Production agents converge on "persistent goal + living plan." Naive recursion and frozen DAGs both fail.

Across the production agent landscape, no single approach to multi-day goal decomposition has converged, but a clear hierarchy has emerged. Pure upfront task trees (BabyAGI's original design, classic HTN) have largely been abandoned for open-ended goals because the plan goes stale within hours. Pure iterative "next step only" approaches (early AutoGPT) suffer from documented infinite loops and "hallucination loops" where the agent re-plans without progressing. The dominant production pattern is now a hybrid: a persistent goal object with a coarse plan that the agent revises continuously, executed by a durable scheduler. Devin 2.0 explicitly describes its preliminary plan as "a starting point, not a contract" that is "revised continuously" by a Universal Planner trained for long-horizon tasks; Manus AI persists the plan as a live todo.md checklist injected into context every iteration.

The 2025-2026 academic literature has been blunt about why naive approaches fail. The Zylos and arXiv "Goal Drift" reports show every frontier model drifts as context grows; Claude 3.5 Sonnet held for ~100k tokens, GPT-4o mini drifted at all lengths. The mechanism is pattern-matching: the deeper the agent gets into its own trajectory, the more it imitates recent local behavior instead of the original system prompt. Why Reasoning Fails to Plan (2026) formalizes this as step-wise greedy myopia. The proposed fix that keeps recurring is structural separation of planning and execution into different inference passes (Plan-and-Act achieves 57.58% on WebArena-Lite vs. <25% for monolithic agents), so the planner's context never gets diluted by execution noise.

Production systems borrow heavily from durable-workflow engines. Temporal's "ambient agents" pattern uses Schedules to trigger a long-running Workflow instance per goal; the Workflow persists state across crashes and accepts Signals and Queries without losing memory. OpenAI's Agents SDK integrates with Temporal and Restate explicitly for "human-in-the-loop tasks" and "long-running tools that return a pointer." This decouples when the agent runs from what it remembers, which is the failure mode AutoGPT had.

Voyager's contribution is subtle but important: it doesn't plan a multi-day goal upfront — it runs an automatic curriculum that proposes the next task based on current capabilities and recent failures, then stores winning code in a skill library indexed by embeddings. This is "iterative refinement plus accumulated experience" and outperformed ReAct, Reflexion, and AutoGPT. For knowledge work, the analog is: don't commit to a 14-day investigation plan on day 1 — persist a goal object, plan the next session, and grow a library of successful research moves.

02Options — pick one

From upfront DAG to pure iteration to event-driven — the production winner is in the middle.

Option 1

Frozen Upfront DAG

Agent writes a complete task tree at goal creation, then executes nodes in dependency order with minimal re-planning.

On goal acceptance, the agent calls a planner model that emits a full DAG of sub-tasks with deadlines, dependencies, and acceptance criteria. The scheduler walks the DAG; each scheduled wake-up just executes the next ready node. Re-planning is rare and gated (e.g., only when a node fails twice).

Pros
  • Predictable budget, easy to estimate cost upfront
  • Simple human review surface — the user approves the whole plan once
  • Resistant to step-wise greedy drift because the plan is fixed
Cons
  • Plans go stale fast for open-ended goals like "monitor churn risk"
  • Cannot incorporate discoveries (a competitor pivots, a customer cancels)
  • HTN-style: works well for procedural domains, poorly for research synthesis
Option 2

Pure Iterative Refinement

Agent plans only the very next step at each wake, with no persistent plan beyond the current scratchpad.

On each wake-up, the agent reads the goal statement, looks at memory of past sessions, and decides one action. After execution, it writes a brief reflection and ends the session. There is no plan object — the "plan" exists implicitly in the goal and accumulated memory.

Pros
  • Maximally adaptive to new information
  • Cheap per wake-up; no planner overhead
  • Matches Voyager's curriculum-style success on open-ended exploration
Cons
  • Documented to produce infinite loops (AutoGPT/BabyAGI failure mode)
  • No user-visible progress structure — hard to do weekly review
  • Severe goal drift risk at long horizons; agent re-derives intent each time
Option 3

Persistent Goal Object with Living Plan

Recommended

A first-class Goal entity holds intent, sub-goals, check-ins, and budget; the agent maintains a living plan (Manus-style todo.md) that it revises each session.

Creating a multi-day goal instantiates a durable Goal record with: intent, success criteria, sub-goals (revisable), schedule (cron or event triggers), budget cap, and a sequence of check-in milestones. Each scheduled wake runs a Plan-and-Act split: a small Planner pass re-reads the Goal and decides whether to keep, amend, or rebuild the sub-goal list; an Executor pass performs one or two actions against the current sub-goal. The Goal object is the source of truth across sessions, not the conversation history.

Pros
  • Structurally prevents goal drift (Planner pass re-grounds on Goal each wake)
  • Natural fit for human check-ins (weekly review = read the Goal object)
  • Composes with durable schedulers (Temporal Schedules, Restate) for crash-safety
  • Mirrors Devin 2.0's "plan as starting point, revised continuously" pattern
Cons
  • More schema design upfront — Goal, SubGoal, CheckIn, Budget entities
  • Planner pass adds per-wake cost (mitigated by caching the Goal snapshot)
  • Requires discipline to keep Goal small enough to fit in Planner context
Option 4

Event-Driven Reactive Plan

No fixed schedule — the agent wakes only when external events (inbox, calendar, webhook) match Goal-attached triggers.

Each Goal registers triggers (e.g., "new email from competitor's PR team," "monthly first-of-month tick," "churn-score change > 10%"). The harness's event bus invokes the agent only on match. The agent has a thin plan but no proactive schedule.

Pros
  • Zero wasted wakes; cost scales with actual signal
  • Reactivity matches real knowledge-work cadence (research is bursty)
  • Avoids the "scheduled but nothing to do" failure mode
Cons
  • Misses goals that need proactive synthesis (no event ever fires for "draft the report")
  • Trigger configuration becomes its own complexity surface
  • Harder to enforce budget caps when wake frequency is unpredictable
Recommendation

Option 3 — Persistent Goal Object with Living Plan.

For a knowledge-work harness whose stated novel primitives already include layered vector memory, proactive scheduler with budget caps, source-provenance lineage, and draft-as-artifact, Option 3 is the only one that composes cleanly with all four. The Goal object becomes the join key for memory (sessions tagged by Goal), the unit the scheduler budgets against, the root of provenance lineage (every claim traces to a Goal-session-source chain), and the parent of draft artifacts. Pure upfront DAG or pure iterative approaches force you to bolt these primitives onto either a static structure or nothing at all. The Devin 2.0 / Manus convergence on "preliminary plan + continuous revision + persistent todo" is not coincidence — it is the production-validated answer to the goal-drift and infinite-loop failure modes the 2025-2026 literature documents.

Implement it with a Plan-and-Act structural split (cheap Planner pass that only sees the Goal + last check-in summary; Executor pass that sees full session context) to inherit the empirical drift-resistance of separated-context architectures. Add hard guardrails lifted from the AutoGPT post-mortems: a max_replan_attempts per wake, a budget cap that hard-stops execution (not just warns), and mandatory human check-ins at user-configured intervals where the agent surfaces the current sub-goal list, recent findings, and proposed next actions. Treat check-ins as first-class Goal events, not chat messages. Finally, borrow Voyager's skill-library idea: when a sub-goal pattern succeeds, store it as a reusable method indexed in vector memory.
