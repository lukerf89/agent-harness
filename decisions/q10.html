Q10 / 12 Memory · Multi-tenant

Single-user, team-shared, or multi-tenant from day zero?

Why it matters Multi-tenancy retrofit is essentially a rewrite. But over-designing for it adds substantial complexity that may be wrong for a solo-knowledge-worker use case. The data-model decision sets the upper bound on what the harness can ever be — pick wrong and either the prosumer-to-team gap kills you, or enterprise machinery swamps the four novel primitives.
01Research findings

The market splits bottom-up (solo→team) vs top-down (enterprise→solo). Dual-tier memory is the convergence.

The market splits cleanly along a bottom-up vs top-down axis. Mem.ai, Mem0, and Granola began solo-first and bolted team workspaces on later — Mem.ai's roadmap explicitly describes the "Mem Graph" as a personal knowledge graph that will extend to teams, and Granola only added shared folders and an enterprise API after the personal note-taker reached scale. Glean and Microsoft 365 Copilot started enterprise-first, with permissions-aware RAG as the foundational primitive — Glean indexes every document alongside its source-system ACL, refreshes via webhooks, and never overrides upstream permissions. The bottom-up products treat memory as an extension of one user's brain; the top-down products treat memory as a federated view over many users' permissioned data.

Across all team-capable products, a dual-tier memory model is converging: each user has private memory the team cannot see, plus a project- or workspace-scoped shared memory. ChatGPT Teams formalizes this most clearly — once a Project is shared, its memory is automatically and irreversibly switched to "project-only," and it cannot read individual users' personal memories or custom instructions. Notion AI takes the inheritance route: a personal agent has exactly the user's page permissions, and team Q&A only sees pages the asker is a member of. Microsoft 365 Copilot stores per-user memory in hidden folders inside that user's Exchange mailbox, inheriting Exchange's compliance posture rather than building a new ACL layer.

On infrastructure, namespace-per-tenant in a shared vector index has become the default 2024-2026 pattern. Pinecone explicitly recommends namespaces over per-tenant indexes — they scale to hundreds of thousands of tenants per index, and querying a 1 GB namespace costs 1 RU vs ~100 RUs for metadata-filtered scans over a 100 GB shared namespace. Postgres RLS shows the same pattern on the relational side: a tenant_id UUID on every row plus policies bound to a SET LOCAL session variable. The known retrofit cost for both patterns is steep but bounded — if the application code consistently passes a tenant context. Retrofitting without a tenant abstraction in the code path is the part that becomes a rewrite.

The conflict-resolution problem ("two PMs decided different things") is largely unsolved in shipped products. Mem0's approach is single-pass ADD-only — memories accumulate, nothing is overwritten — and arbiter LLMs decide at retrieval time whether a new statement is a duplicate, refinement, or pivot. Enterprise memory research increasingly argues memory must store who decided, when, and on what authority rather than collapsing to a single "current truth." This dovetails directly with the harness's existing source-provenance lineage primitive.

02Options — pick one

From "solo, schema-ready" to "enterprise on day one" — the recommended middle inherits provenance for free.

Option 1

Solo-only v1, schema-ready for teams

Ship single-user memory now, but lay down user_id + org_id columns, namespace-per-user vector storage, and a tenant-context middleware from day zero.

Every memory row carries (org_id, user_id, scope) even when org_id == user_id. Vector chunks live in a per-user namespace. All read/write paths go through a MemoryContext abstraction that takes a tenant. No ACL UI, no sharing UI, no team features — but the wiring exists. When teams ship in v2, you add a team scope, RLS policies, and a sharing UI without touching the storage layer.

Pros
  • Solo prosumer is the proven beachhead (Mem.ai, Granola, Mem0 all started here)
  • Avoids ACL, conflict, and discovery UX work that would distract from novel primitives
  • Privacy is trivially correct — there is no other user
Cons
  • "Shared decision discovery" use case is unbuildable in v1
  • Solo-only products struggle to "expand" into companies later
  • Postpones the hardest design question (cross-user conflict + provenance) past the point where the data model is set
Option 2

Team-from-day-zero with permissions-aware retrieval

Build a Glean-style permissions-aware memory layer up front — every memory chunk carries an ACL inherited from its source document.

Adopt the Glean model: when memory is written, capture the source doc's ACL (Drive permissions, Gmail recipients, Slack channel members) alongside the embedding. Every retrieval filters by the asking user's identity against the chunk ACL. Conflict resolution stores all perspectives with provenance and surfaces conflicts to the user. Solo users get a degenerate single-member team.

Pros
  • "What did anyone decide about Acme?" works on day one
  • Source-provenance lineage becomes the natural carrier for ACLs
  • No retrofit risk; the hardest architectural decision is made before scale
  • Aligns with where Microsoft Copilot, Glean, and Notion AI are converging
Cons
  • Substantial complexity tax on v1: ACL inheritance, real-time permission updates, conflict UI, identity model
  • Solo users pay for machinery they don't need, slowing PMF iteration
  • Conflict-resolution UX is unsolved territory — you'll be inventing
Option 3

Federated personal memory with explicit shared artifacts

Recommended

Each user owns a private memory namespace; shared knowledge lives in explicitly-promoted "shared artifacts" with their own ACL.

The ChatGPT Teams Projects + Granola Shared Folders model. Personal memory is solo-only, fully private, never queryable by anyone else. To share, a user promotes a memory chunk, a draft, or a research note into a "shared workspace" with explicit team ACL. The agent reads union(personal, accessible-shared) at query time. Provenance always identifies which workspace a chunk came from. No automatic cross-user leakage is even possible.

Pros
  • Matches the dual-tier model the market is converging on
  • Privacy is structurally enforced — CEO emails in personal memory cannot leak
  • Conflict question simplifies: personal memories never conflict with peers'; shared artifacts conflict only when explicitly co-edited
  • Promotion gesture creates a clean audit trail that pairs with provenance lineage
Cons
  • "What did anyone decide about Acme?" only works for explicitly promoted artifacts
  • Requires inventing a "promote to shared" UX pattern early
  • Still needs workspace_id plumbing on day zero to avoid the retrofit cliff
Option 4

Full multi-tenant SaaS from day zero

Treat the harness as a B2B SaaS from commit one — orgs, roles, audit logs, SCIM, RLS, the lot.

Standard multi-tenant Postgres with RLS on every table, Pinecone namespace per org with metadata filters per user, role-based memory visibility, admin console, audit logs.

Pros
  • Zero retrofit risk
  • Enterprise-ready from day one
Cons
  • Enormous up-front complexity tax that will swamp the four novel primitives
  • Solo users (your likely first 100) get a worse product than Mem.ai
  • You'll spend the first year building SaaS infrastructure rather than the agent
Recommendation

Option 3 — Federated personal memory with explicit shared artifacts.

The knowledge-work harness's four novel primitives are easiest to validate on a solo user. But the draft-as-first-class-artifact primitive already implies an artifact graph that can carry an ACL — drafts have authors, reviewers, and recipients regardless of whether the agent is solo or team. Reusing that artifact spine for "shared memory chunks" is essentially free, and it neatly sidesteps the worst question — "whose memory wins when two PMs decided different things" — by making personal memory non-conflicting by construction and confining conflicts to artifacts where humans already have collaboration norms (comments, versions, merges).

Concretely: ship with (workspace_id, user_id) on every row from commit one (cheap insurance against the Option 1 retrofit), Pinecone namespace per user, RLS policies that gate by workspace and user, and a workspace_id == user_id "personal workspace" as the v1 default. Don't build sharing UI in v1, but do build the schema so v2 can add a "promote to team workspace" gesture without a migration. This gives you Mem.ai's solo speed of iteration with Glean's structural privacy guarantees, and it positions provenance lineage to carry ACL metadata when teams arrive — turning what would be a rewrite into a feature addition.
