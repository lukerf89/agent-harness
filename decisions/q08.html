Q08 / 12 Drafts · Concurrency

How should the draft state machine handle user edits during agent revision?

Why it matters Without a model here, the draft state machine becomes "whichever side wrote last wins," which silently destroys work. CRDTs auto-merge cleanly at the syntactic level but produce semantic nonsense when the user's sentence interleaves into an AI-generated paragraph. The shipping consensus is clear — and counter-intuitive.
01Research findings

Every shipping AI editor reached the same conclusion: AI doesn't directly mutate user-authored text.

The shipping consensus across AI editing tools in 2025-2026 has converged on a clear pattern: AI does not directly mutate the user's authored text in-place during generation. Instead, AI output is materialized as a reviewable artifact — ghost text, suggestion overlay, diff view, or branched candidate — that the user explicitly accepts, rejects, or partially merges. This is true across radically different domains: GitHub Copilot dismisses ghost-text inline suggestions the moment the user keeps typing; Notion's "Suggested edits" mode wraps changes in underlined/strikethrough markup until accepted; Cursor's Apply model runs a specialized model whose entire job is to translate a frontier model's rough sketch into a clean diff; Tiptap's AI Suggestion and AI Changes extensions render edits as a diff view; Google Docs' Gemini "Help me write" surfaces tone-changes as bulk Accept all / Reject all; Linear's Triage Intelligence proposes duplicates and labels with keyboard-shortcut accept/decline.

CRDTs are widely deployed for the human-human layer (Notion, Google Docs, Tiptap collaboration, Yjs ecosystem) and Yjs in particular is the de-facto winner for performance and ecosystem. However, the Ink & Switch / Yjs community itself acknowledges a hard limit: "Plain text CRDTs only preserve low-level syntactic intent" — a CRDT will happily interleave a user-typed sentence into the middle of an AI-generated paragraph, producing syntactically valid but semantically garbled prose. Electric SQL's April 2026 architecture for "AI agents as Yjs peers" only works because the agent uses constrained, human-like tool calls scoped by relative anchors, not free-form text replacement.

Cursor 2.0 (Oct 2025) shipped what is essentially the industrial reference design for the harder case (large agent-generated changes): isolated git worktrees per agent, up to 8 parallel agents, aggregated diff view, per-agent undo, and an Apply step that merges back into the working branch — or discards the entire worktree if rejected. Lovable's 2.0 GitHub two-way sync follows the same pattern. Granola's design is the canonical knowledge-work analog: user notes stay black, AI enhancements render in grey with timestamp-linked provenance, the user's structural skeleton is preserved across re-enhancement, and "raw" vs "enhanced" are explicitly separate views.

Locking is almost universally avoided. The arXiv 2025 paper on collaborative editing with users and AI agents found optimistic concurrency wins over pessimistic locking for fluidity; the only durable use of locking is field-level or display-pagination locking (to prevent agent edits from yanking the user's scroll position) — never document-wide blocking.

02Options — pick one

Each shipped product has all three modes hiding inside it. The recommendation is to make that explicit.

Option 1

Yjs CRDT with agent as constrained peer

Run the agent as a server-side Yjs peer that issues human-like, anchor-scoped operations rather than free-form rewrites.

The draft is a Yjs document. Both the user's editor and the agent connect as peers. The agent never replaces a whole paragraph blindly; it issues bounded operations (insert at relative anchor, replace selection range identified by relative position). Yjs auto-merges concurrent edits via Peritext/YATA semantics. The agent's presence is shown via awareness state.

Pros
  • Zero blocking — user keeps typing while agent works
  • Reuses battle-tested Yjs ecosystem (fastest CRDT, mature editor bindings)
  • Durable across disconnect (agent runs server-side)
  • Plays well with the harness's multi-session model
Cons
  • CRDTs preserve syntactic, not semantic, intent — interleaved edits can produce nonsense
  • Requires careful agent prompt-engineering to use constrained tool calls
  • Hard to attach provenance to specific spans
  • Conflict invisibility — silent semantic clobbering is worse than a visible conflict
Option 2

Agent-as-suggestion (tracked changes / diff view)

Agent never directly mutates the canonical draft — every agent output is a proposed delta the user accepts, rejects, or partially merges.

Model the draft like Notion's "Suggested edits" or Tiptap's AI Suggestion: AI revisions are persisted as a suggestion layer over the base document. The user keeps typing freely on the base; suggestions render as overlay (inserts in colored underline, deletions struck-through). The user accepts per-suggestion or accept-all. The user's concurrent typing never conflicts because they edit different layers.

Pros
  • Matches the shipping consensus across Notion, Tiptap, Google Docs, Copilot, Cursor
  • Provenance is trivially attached at the suggestion level
  • Aligns with the harness's "draft as first-class versioned artifact" — every accept is a versioned event
  • User retains full agency; no semantic surprises
Cons
  • Cognitively heavier for large rewrites — accepting 200 suggestions is annoying
  • Requires building suggestion-layer infra (or adopting Tiptap AI Suggestion / Liveblocks)
  • Stale suggestions when user edits the underlying span — needs invalidation
  • Less natural for "agent autonomously drafting from scratch"
Option 3

Fork-and-merge (worktree model)

Agent revisions happen on a branch; the user merges the candidate back, manually or with model-assisted three-way merge.

Mirror Cursor 2.0's worktree model. When the agent revises, it forks the current draft into a candidate version. The user keeps editing the main draft live. When the agent finishes, the user sees an aggregated diff (base vs candidate vs current-main). They Apply, partial-apply per section, or discard. Conflicts trigger a model-assisted three-way merge presented as a resolution UI.

Pros
  • Perfect for large agent revisions that touch most of the doc
  • Clean isolation — user never sees mid-generation flicker
  • Natural fit for proactive scheduler: agent runs revisions in background
  • Each branch is a first-class versioned artifact with provenance
Cons
  • Overkill for small inline tweaks ("fix grammar in this sentence")
  • Merge UI is the hardest piece to design well — Cursor took years
  • Three-way merge of prose is genuinely hard; LLM-assisted resolution can hallucinate
  • More latency from user POV — no real-time co-editing feel
Option 4

Hybrid tiered model (inline / suggestion / branch)

Recommended

Pick the merge mode based on edit scope — inline ghost-text for small completions, suggestion-layer for medium revisions, branch-and-merge for large rewrites.

The agent classifies (or the harness routes) its intended edit into one of three modes. Tier 1 — token-to-paragraph inline completion uses Copilot-style ghost text dismissed on keystroke. Tier 2 — section-level revision ("improve this paragraph", "make more formal") uses Notion/Tiptap suggestion layer with per-span accept. Tier 3 — whole-document or multi-section regeneration uses fork-and-merge with diff review. Underneath, the document is a Yjs CRDT for the human-collaboration layer (multi-user, multi-device).

Pros
  • Matches the empirically-shipping pattern — nobody uses one mode for everything
  • User cognitive load proportional to change size
  • Each tier produces appropriate provenance granularity
  • Composes cleanly with all four novel components
Cons
  • Most engineering surface — three UX modes to build and maintain
  • Tier classification can be wrong, leading to mismatched friction
  • More state machine complexity in the harness
  • Harder to explain to users initially
Recommendation

Option 4 — Hybrid tiered model.

Every shipping product converges here in practice, even if they don't articulate it that way: Cursor has Tab (ghost text) + Composer inline edits + Composer worktrees; Notion has slash-AI insert + Suggested edits + AI blocks; Google Docs has smart compose + Help-me-write suggestions + side-panel rewrites. The reason is straightforward — semantic merge difficulty scales with edit size, and forcing every edit through one mechanism either over-burdens small edits (every comma fix is a PR) or under-protects large ones (a 500-word rewrite silently CRDT-merges into nonsense). For a knowledge-work harness where the agent is genuinely autonomous and may revise drafts proactively while the user is offline, the worktree-style Tier 3 is non-negotiable; for inbox triage and quick correspondence, Tier 1/2 inline UX is what makes the tool feel responsive.

Build it concretely as: Yjs as the substrate for the human-collaboration and presence layer (it's the obvious winner — fastest, biggest ecosystem); Tiptap AI Suggestion / Liveblocks-style suggestion overlay for Tier 2; a lightweight branch model (does not need real git — just versioned document snapshots with parent pointers, plus an LLM-assisted three-way merge resolver) for Tier 3. Crucially, never block the user's editor during agent generation — every tool that tried locking lost. Pair this with the harness's provenance lineage by attaching source citations at the suggestion or branch level (not at CRDT op level — that granularity is wrong), and the draft-as-versioned-artifact primitive maps directly onto Tier 3 branches plus a journal of accepted Tier 2 suggestions.
