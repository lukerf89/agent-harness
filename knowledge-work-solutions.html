01The gaps, restated

Four from the design report, four more from the same pressures.

The original design report named four gaps that the five studied coding harnesses don’t solve. Re-reading it against the 2026 OSS landscape, four more surface — not because they were missed, but because each is an obvious consequence of the same three architectural asymmetries (verification, blast radius, source-of-truth). They're called out below in gold; the originals in red.

Gap 1 · Design report

Retrieval-augmented memory across sessions

Vector store, semantic recall, provenance on every fragment, staleness signal — none of the five harnesses do this; they lean on filesystem reads.

Pick Graphiti + Qdrant + bge-v2-m3
Gap 2 · Design report

Proactive, long-horizon loops

Cron + webhook + RSS scheduling, days-to-weeks suspensions, budget caps, idempotent re-wakes — all five harnesses are reactive only.

Pick Restate
Gap 3 · Design report

Source provenance surviving synthesis

Every claim carries traceable lineage to its underlying sources, even after multi-hop summarization. Trivial for coding (the code is the source); unsolved for knowledge work.

Pick Citations API + Instructor + custom DAG
Gap 4 · Design report

Draft / refine cycle as first-class state

A draft isn't a message turn; it's a versioned document with author attribution, send-binding to specific versions, and structured critique input.

Pick Yjs 14 + BlockNote + DSPy Refine
Gap 5 · New

Observability and continuous eval

Verification is human and slow — the compiler doesn’t save you. So you instrument every step and run eval against production traffic, not just at build time.

Pick Langfuse + DeepEval + Inspect + Argilla
Gap 6 · New

Document & multimodal understanding

Knowledge work is PDF-shaped, slide-deck-shaped, spreadsheet-shaped, meeting-audio-shaped. Coding harnesses barely touch any of it.

Pick Docling · PaddleOCR-VL · GMFT · ColQwen · WhisperX
Gap 7 · New

Tool / MCP ecosystem at scale

Coding agents need ~10 tools; a knowledge worker’s ~100. The challenge isn’t per-tool; it's tool selection, multi-tenant OAuth, naming dedup, and not blowing the context window.

Pick Nango + Code Mode + FastMCP + Obot
Gap 8 · New

Engine: build vs adopt

Every framework wants to be your engine. The five studied harnesses prove you can roll your own. What primitives are actually worth borrowing in 2026?

Pick Build it. Borrow three things.
GAP 01MEMORY · RETRIEVAL · STALENESS

The repo is not your substrate any more.

Design report

The coding-harness memory model is a sliding window over an authoritative substrate (the git repo). Take the substrate away — which is exactly what knowledge work does — and what remains is lossy compression of the only copy. The third tier of memory (cross-session, semantic, provenance-aware, staleness-aware) is the single largest architectural gap in the field.

Recommendation
Build on Graphiti (Apache 2.0, bi-temporal knowledge graph by Zep) as the primary memory layer, with Qdrant (Apache 2.0) as the parallel vector store for content that doesn’t deserve a graph node, and bge-reranker-v2-m3 via AnswerDotAI/rerankers for the rerank step. The bi-temporal data model (valid_from, valid_to, ingested_at on every edge) is the only OSS architecture where provenance, staleness, and contradiction tracking are first-class — not bolt-ons.

Mem0 is the benchmark king (LoCoMo 91.6, LongMemEval 94.8) and the loudest in the space. It would be the obvious pick if the bar were “turnkey embeddings + entity linking.” The bar isn't that — it's defensible provenance. Mem0's v3 went single-pass ADD-only by design, optimizing throughput at the cost of explicit supersession with audit trail. The mem0 team's own “State of Agent Memory 2026” post lists staleness in high-relevance memories as an open problem. That's the disqualifier.

Graphiti pays for its provenance with extraction cost — the mem0 paper measured Zep's token footprint at ~600k per conversation versus mem0's 1,764. Budget for it, batch ingestion off the hot path, and accept the tradeoff. You also pay with a graph backend dependency (Neo4j 5.26+, FalkorDB, Kuzu, or Neptune); Kuzu is the lightest if you want to avoid running Neo4j.

How it slots into the three-tier model

  • Working context — your loop owns it; no library.
  • Session memory — append-only log per session in Postgres. Also write each utterance as an episode into Graphiti, which derives entities/edges asynchronously.
  • Persistent memory — two parallel reads on every turn: Graphiti query for typed facts with validity windows (filter valid_to IS NULL for “currently true”) plus Qdrant hybrid query over a chunks collection where payload carries session_id, source_uri, speaker, stated_at, superseded_by. Fuse candidates, rerank top ~50 with bge-v2-m3, take top ~8 into the prompt.

Staleness becomes first-class: if a recalled fact's valid_from is >N months old and its category is volatile (deadlines, statuses, preferences), the harness prepends [STALE: last confirmed YYYY-MM-DD] before handing the chunk to the model. Graphiti gives you the timestamps; the harness owns the staleness policy.

Still unsolved

Cross-modal memory. Graphiti and mem0 are both text-first. A meeting transcript with embedded slides, a Slack thread with screenshots, a doc with tables — none of these have a clean memory schema. You'll model multimodal nodes yourself.

GAP 02DURABLE EXECUTION · PROACTIVE LOOPS · HITL

Reactive loops are the wrong primitive.

Design report

The five coding harnesses are all reactive: user prompts, agent runs, agent stops. Knowledge work needs proactive — wake on a schedule, wake on a webhook, suspend for days awaiting an approval, retry idempotently, fail under a budget cap. None of that lives inside an agent SDK. It lives in a durable execution engine sitting underneath one.

Recommendation
Restate (Apache 2.0, single Rust binary). The awakeable primitive is the cleanest HITL token in any durable engine — the agent creates one, hands the ID to an email/Slack/webhook, durably suspends, and resumes when something resolves or rejects. Skip Inngest (SSPL is hostile to embeddable harnesses), skip Temporal (overkill unless you're already running it), skip the LangGraph server (Elastic-licensed trap). DBOS is the runner-up if you're fully Postgres-coupled.

The honest reason to skip Temporal for a new knowledge-work harness: it requires you to rearchitect the app around Workflow/Activity boundaries, run a separate cluster (or pay Temporal Cloud), and write >100 LoC of integration where DBOS needs ~7. For a coding-agent harness team at FAANG scale, sure. For a knowledge-work harness that needs to ship and be self-hostable by users, it's overkill.

Restate wins because the awakeable model maps directly to the dominant HITL pattern in knowledge work: “ask the user to confirm before sending; wait indefinitely.” The agent creates an awakeable, hands the ID to whatever surface will collect the answer (a Slack DM, a web modal, an email), suspends the workflow, and resumes — same place, same state — when someone resolves the token. No polling. No process kept alive. Days of waiting cost zero compute.

What the engine doesn't give you (build these)

  • Per-loop budget ceiling. Wrap every LLM/tool call in a BudgetTracker keyed by loop-id; on overage, durably suspend and signal a human. 2026 consensus: hard caps for free tier, soft caps with model-downshift (Opus → Haiku) for paid.
  • Side-effecting idempotency. Durable execution gives exactly-once step semantics, but only for steps the engine knows about. The instant your agent sends an email, the engine can't unsend it. Write-through idempotency cache keyed by (loop_id, tool_name, content_hash) inside every side-effecting tool, plus an LLM-level “did I already do this” check.
  • Goal-updating semantics. The agent's stated objective evolves over weeks. Model goals as a separate versioned entity (one row per revision), pass (goal_v_n, evidence_so_far) into each wake, and treat agent-proposed goal changes as HITL approvals via awakeable.
  • Wake-up scheduling beyond cron. Webhook ingestion translates external events into awakeables; polling workflows need the same budget and idempotency treatment, because an uncapped poller is exactly the $400-overnight failure mode.
Still unsolved

Goal coherence under indefinite suspension. A workflow paused 14 days for human approval may resume into a world where the goal has shifted, the data has aged, the user's preferences changed. No engine handles this — you'll write a re-grounding step that runs on every resume.

GAP 03CITATIONS · LINEAGE · MULTI-HOP SURVIVAL

Provenance has to survive the second summarization.

Design report

Every popular framework treats citation as a property of the final answer, not a property that propagates. Summarize a summary and every one of them breaks. This is the cleanest hole in the OSS landscape: multi-hop synthesis provenance is genuinely unsolved. The grounding-API and structured-output pieces exist; the lineage layer doesn't.

Recommendation
Composite stack with no single owner. Anthropic Citations API (or Cohere Command R+ fallback) at the grounding layer. Instructor's CitationMixin for typed citation enforcement at every synthesis step — its Pydantic validator verifies quoted strings actually exist in the source. Vercel AI SDK Sources + shadcn ai/inline-citation for the UI rail and inline markers. The multi-hop lineage DAG you'll model yourself, taking Flowcept / PROV-O (MIT, research-grade) as the schema reference.

The architecture you'll end up building has four layers stacked: grounding (the model emits structurally-cited tuples), schema (every agent step returns Pydantic with a parent_citations: list[CitationRef] field, declaring which input citations each step consumed), lineage DAG (a PROV-O-shaped graph where retrieved facts are prov:Entity nodes and synthesis steps are prov:Activity nodes), and UI (rendered citation markers with cross-hop ancestry navigation). Three of the four are off-the-shelf; the lineage DAG is ~200 lines of custom code modeled on Flowcept's schema.

The query “what original entities support claim X” becomes a DAG traversal that survives N synthesis hops. Nobody ships this. Multiple 2025-2026 research papers (Wang et al. CHI 2026, ContraGen, DRAGged-into-Conflicts) confirm the gap is real. This is where your harness gets to plant a flag.

The composition pattern in practice

class SynthesisStep(instructor.CitationMixin, BaseModel):
    claim: str
    quoted_evidence: list[str]   # validated to exist in source
    parent_citations: list[CitationRef]  # which prior step's citations this depends on

# At every hop, the schema forces you to declare lineage.
# At read time, you DAG-traverse parent_citations back to the
# original retrieval entities, surviving arbitrary synthesis depth.
Still unsolved

Inter-source contradiction handling. ConflictRAG and Micro-Act (2025-2026 papers) identify knowledge conflict between retrieved sources as the next frontier. No production OSS framework handles it. You'll build a contradiction-detector on top of the lineage DAG and either surface conflicts to the user or route them to a critique sub-agent for resolution.

GAP 04DRAFTS · VERSIONS · CRITIQUE · ATTRIBUTION

A draft is a Yjs document, not a chat message.

Design report

The single most important architectural commitment for draft-refine: a draft is a CRDT document, not an entry in message history. Versions are snapshots in that doc; “send” binds to a specific snapshot ID. Every other piece of the system flows from this choice. The Yjs ecosystem is the only one mature enough to ship today, and Yjs 14 is landing the attribution primitives that close the human-vs-agent gap.

Recommendation
Yjs 14 (changesets + attributions, landing in 2026) as the CRDT layer. BlockNote as the editor (Notion-style block-based, already shipping the AI extension @blocknote/xl-ai with accept/reject suggestion UI); TipTap as the escape hatch if you need deeper control. Hocuspocus for the sync server. DSPy Refine v3 for the critique loop, wired to span-anchored editing tools (the Electric.ax “agent as CRDT peer” pattern). Yjs encodeSnapshot for versions; bind send to a specific snapshot ID. Skip Loro and Automerge — both better in some ways, neither ecosystem-mature.

The reason to commit to Yjs despite Loro being technically superior on interleaving guarantees: Yjs 14's changesets + attributions land in 2026, and BlockNote is actively driving the spec in collaboration with the Yjs maintainer. That's the missing primitive for human-vs-agent attribution, and it's landing in the dominant ecosystem. Revisit Loro in 2027.

The critique loop should follow the Electric.ax “agent as CRDT peer” pattern: critics don't regenerate the document — they emit Yjs ops anchored to CRDT items via span-based tools (search_text returning stable match handles, place_cursor with relative positions that survive concurrent edits, start_streaming_edit for incremental insertion). DSPy Refine wraps the loop with rubric enforcement; the editor shows pending agent edits with accept/reject affordances.

The version + send binding model

// Postgres
draft (
  id, yjs_doc_id,
  current_snapshot_id,    // moves forward continuously
  ratified_snapshot_id,   // bound when user clicks "Send"
  ratified_at, ratified_by, ...
)

// On Send: capture Y.encodeSnapshot(doc), bind to send action,
// require re-confirmation if current_snapshot_id != ratified_snapshot_id.
// Same shape as a git tag — cheap, principled, auditable.
Still unsolved

Semantic attribution of mixed-author content under refactoring. Yjs 14's attributions tell you who originally inserted a character. When an agent rewrites a paragraph the human partially wrote, attribution either shatters per-character or collapses to “the rewriter.” Multi-agent conflict resolution (two critics editing overlapping spans) is also unhandled — the CRDT will happily merge into nonsense. No agent-aware merge policy exists in OSS. This is the remaining 30% gap after Yjs 14 closes the first 70%.

GAP 05OBSERVABILITY · EVAL · ANNOTATION

The compiler isn't going to save you, so instrument everything.

New gap

The verification asymmetry from the design report has a direct consequence the original missed: without ground truth, you cannot ship without comprehensive eval and observability. Coding harnesses get away with thin telemetry because the tests are the eval. Knowledge work has nothing equivalent — instrument every step, run online evals against sampled production traffic, and budget for a continuous annotation surface for domain experts.

Recommendation
Langfuse (MIT, MIT-retained post-ClickHouse acquisition) as the primary tracer + storage. Langfuse's built-in evals + DeepEval for online sampled scoring. Inspect AI (UK AISI) for offline agent-trajectory regression. promptfoo for prompt CI on every PR. Argilla for dataset curation. Emit pure OpenTelemetry GenAI semantic conventions as the wire protocol — that way Langfuse, Phoenix, OpenLIT, or any OTel backend can all consume the same traces. Skip Phoenix (Elastic 2.0), Helicone (maintenance), AgentOps (stagnant since Aug 2025).

The two-surface eval pattern matters in 2026: online sampled in Langfuse (LLM-judge against 1-5% of traces, flagging drift) plus offline regression in Inspect AI (golden trajectory replays in CI). promptfoo runs the cheap prompt-level CI gate on every PR. The three are complementary, not redundant.

The OpenTelemetry GenAI semantic conventions exited experimental for client spans in early 2026 and are stable in practice for agent spans (create_agent, invoke_agent, execute_tool). Emitting OTel from the harness via OpenLLMetry gets you portability for free — you can swap Langfuse for OpenLIT or any OTel backend without rewriting instrumentation.

The annotation surface — build a custom UI

The reviewer for a knowledge-work harness is a paralegal, an analyst, an executive assistant — not a Python engineer. Langfuse's built-in queue UI assumes the latter. Build a thin custom UI on top of Langfuse's scores API where the reviewer sees only the rendered final artifact + 3-5 rubric questions, no spans, no tokens. Langfuse themselves published the pattern in November 2025.

Still unsolved

Evaluating non-verifiable outputs (“is this draft worth the user's time to edit”) is genuinely hard. Off-the-shelf evals score generic axes — faithfulness, hallucination, toxicity. None evaluate writing voice, contextual tone-fit, or strategic intent. You'll build domain rubrics, validate them against human grading regularly (judge drift is real), and probably maintain a small panel whose agreement rate IS your eval ground truth. Also unsolved: counterfactual replay across trajectories where prompt changes mutate tool calls.

GAP 06DOCUMENT · MULTIMODAL · AUDIO

Knowledge work is PDF-shaped. The agent needs to read it.

New gap

Coding agents touch the filesystem and call it done. Knowledge work runs on board decks, contracts, research papers, SEC filings, scanned forms, meeting audio, spreadsheets. The document-understanding stack is its own subsystem — and 2026 is the first year permissively-licensed OSS has closed enough of the gap with commercial APIs to be the default.

Recommendation
A tiered pipeline, not a single parser. Docling (Apache 2.0, IBM-backed, Linux Foundation donation) as the primary: produces DoclingDocument with bbox provenance, native PPTX/DOCX/XLSX/PDF/HTML, hierarchical chunker already wired for RAG, plus the Granite-Docling-258M VLM for the hard pages. PaddleOCR-VL 1.5 (Apache 2.0) on demand for scans and skewed photos — beats GPT-4o on OmniDocBench at 0.9B parameters. GMFT for table rescue. ColQwen2.5 for vision retrieval over figure-heavy corpora. MarkItDown for office files where you don't care about layout. WhisperX for audio. Skip Marker (GPL + cc-by-nc-sa-4.0 weights with $5M revenue waiver = time bomb), Unstructured.io OSS (intentionally de-tuned), and PyMuPDF (AGPL).

Don't ship a single read_document(path) tool. Ship four discrete tools the agent can choose between by inspection:

  • parse_structured(path) — Docling for the 80% case (born-digital PDFs, Office files)
  • parse_with_ocr(path) — PaddleOCR-VL for the hard 20% (scans, photos, skewed pages)
  • extract_tables(path, pages) — GMFT when the parser misses internal cell structure
  • search_visually(query, corpus) — ColQwen2.5 for figure/chart/deck queries
  • transcribe_audio(path) — WhisperX

Crucially: OmniDocBench scores the parsing model, not the pipeline. Docling isn't even on the leaderboard (PaddleOCR-VL and MinerU top the OSS rankings). Docling's value is the end-to-end package — schema-stable DoclingDocument, bbox provenance, hierarchical chunker, an OpenShift operator, native non-PDF formats. That's the right shape for a harness; raw benchmark score isn't.

Cost reality (2026)

Self-hosted production GPU is real money ($2k–5k/month for redundant A100/L40S capacity). Commercial APIs (LandingAI, AWS Textract) run $0.01–0.03/page. The crossover happens around 150k–300k pages/month — below that, paid APIs are cheaper than running your own. Docling on a single L4 with Granite-Docling-258M lands around 1–2 pages/sec; a 200-page filing costs ~$0.02–0.05 on Modal/RunPod.

Still unsolved

Charts/diagrams as data, not images. Every parser extracts a chart bbox + caption; none reliably reconstruct the underlying series. Vision-LLM approaches sit around ~70% accuracy. Slide decks with composed graphics, financial filings with multi-page tables and inline footnotes, handwriting and whiteboard photos, form understanding — all still need either human verification or paid commercial APIs. ColPali-style retrieval is the practical workaround for decks: return the page image and let the agent's VLM reason.

GAP 07MCP · CONNECTIVITY · MULTI-TENANT OAUTH · TOOL SELECTION

The problem isn't the protocol. It's the long tail.

New gap

Coding agents need ~10 tools. A knowledge worker's harness needs ~100, one per SaaS app the user touches. MCP solved the protocol question. The remaining hard problems: long-tail SaaS coverage, multi-tenant OAuth with sane re-consent flows, and not blowing the system prompt by stuffing 100 tool definitions into it.

Recommendation
Nango (Elastic License, self-hosted) for the connectivity layer — 800+ APIs, real multi-tenant primitives, sub-100ms overhead, sync+webhooks+tools in one. Code Mode (Anthropic/Cloudflare pattern) for tool exposure to the agent — empirically 98.7% token reduction, 30-40% latency win, solves the “300 tools in system prompt” disaster cleanly. Semantic tool retrieval as fallback for smaller models. FastMCP 3.x for first-party MCP server scaffolding. Ory Hydra for the harness's own OAuth provider role; Obot or MCPJungle as a self-hosted gateway for allowlisting, audit, and per-tenant scoping. Skip Composio as a primary (the SDKs are OSS but the runtime is closed) — keep Pipedream Connect as fallback for apps Nango doesn't cover.

The biggest architectural shift of 2025-2026 in this space: Code Mode kills the “stuff every tool in the system prompt” pattern. Anthropic's Google Drive → Salesforce benchmark went 150k → 2k tokens with progressive disclosure. Cloudflare's pattern is similar — one sandboxed TypeScript runtime over a generated SDK of all the agent's tools. The whole inbox-triage scenario only works at this scale if the agent can write const urgent = await gmail.search('is:unread').filter(...); await Promise.all(urgent.map(m => linear.createIssue(...))) in one step rather than 30 round-trips.

The Code Mode pattern composes with semantic tool retrieval for smaller models that can't write code well: retrieve top 5 relevant tool definitions by query embedding, expose only those. Multiple gateways (Obot, lunar.dev, StackOne) ship the retrieval layer off the shelf.

Security baseline

The MCP spec still lacks cryptographic auth between client and server (per the 2026 threat-modeling literature). Tool poisoning hits 60-72% success against unmitigated agents on the MCPTox benchmark. Assume hostile tool descriptions; allowlist aggressively at the gateway. For a multi-tenant harness, this is non-negotiable.

Still unsolved

OAuth scope-upgrade UX. User originally granted gmail.readonly; agent now needs gmail.send. No OSS tool handles the re-consent flow gracefully. Per-tenant dynamic tool catalogs driven by user's actually-connected apps — gateways give you static allowlists, not dynamic per-user. End-to-end cryptographic audit trails from chat turn → tool call → upstream API. These are where your harness differentiates; everything else is plumbing.

GAP 08ORCHESTRATION FRAMEWORK · ENGINE

Build the engine. Borrow three primitives.

New gap

Every framework wants to be your engine. The five studied coding harnesses — pi, Claude Code, codex, code_puppy, opencode — all chose to roll their own. That bias is correct for knowledge work too. The harness is the product; if it's someone else's StateGraph, the moat doesn't exist.

Recommendation
Build your own queryLoop. Borrow exactly three things: Pydantic AI's typed Agent + Dependencies + Tools shape (lift the patterns, optionally lift the package), Claude Agent SDK's hook lifecycle taxonomy (PreToolUse · PostToolUse · SubagentStart · SubagentStop · PreCompact · PermissionRequest · UserPromptSubmit), and BAML for all structured output (drafts, citations, synthesis schemas — Schema-Aligned Parsing is the one piece that's genuinely worse to roll yourself). Outsource durability to the workflow engine from Gap 02 (Restate/DBOS). Skip LangGraph wholesale adoption — its checkpointer is too coupled to StateGraph and optimizes for deterministic enterprise compliance flows, the opposite of an LLM-driven agent loop.

The LangChain consultants will tell you to adopt LangGraph wholesale — “the integration surface is too large to roll your own.” That advice is right for enterprise teams shipping commodity refund-approval flows. It is wrong for a knowledge-work harness where the harness is the differentiation. The LinkedIn/Klarna case studies prove LangGraph works for deterministic compliance flows. They don't prove it's the right shape for a research-synthesis or drafting agent — those are much closer to Claude Code's “give the agent a workspace and let it iterate” pattern.

The three borrows in concrete form

  • Pydantic AI's shape — every Agent is typed in its dependencies (DI via RunContext) and outputs (Pydantic model). This is the cleanest typed-Python idiom and integrates with Temporal/DBOS/Restate for durability — exactly the boundary you want.
  • Claude Agent SDK's hook vocabulary — the seven hook names are the right taxonomy. Knowledge work needs PreCompact (drafts get long) and PermissionRequest (send-email-as-me) especially badly. Use the names even if you don't use the SDK.
  • BAML for structured output — for citation schemas, draft schemas, synthesis schemas. Run it as a parallel compile step. It outperforms Instructor and Mirascope on nested structures and smaller models, and Schema-Aligned Parsing handles the dirty reality of LLM output that strict JSON schemas can't.

For sub-agent variants — study three references, build your own

Variants are too central to the product to outsource. Study (1) opencode's two-method dispatch (task() category vs call_omo_agent() direct), (2) Claude Code's worktree-isolated Task tool, (3) fast-agent's Router pattern. Build something that fits your draft/research/critique/summarize variant model from the design report.

Still unsolved

A clean abstraction for the “variant” pattern across frameworks. Every harness reinvents subagent dispatch. There's no agreed-upon primitive (handoffs vs subagents vs routers vs variants vs crews vs swarms vs managed agents). For now: write your own, expect it to be the most opinionated, most-changing piece of your harness, and don't try to make it general.

Build · no one ships this

The harness's actual differentiation.

Nothing in the OSS landscape solves these cleanly. These are the engineering bets that decide whether the harness is a chat interface or a colleague.

  • queryLoop — the AsyncGenerator engine Steering vs follow-up split, yield boundaries, variant dispatch — borrow primitives, write the loop
  • Multi-hop provenance lineage DAG + cross-hop ancestry UI PROV-O-shaped graph, traversal queries, UI for “trace this claim back to original sources” — the cleanest hole in the field
  • Three-tier memory orchestration policy Graphiti gives the timestamps; the harness decides staleness policy, what's volatile vs durable, how to fuse with vector recall
  • Per-loop budget tracker + model-downshift Hard caps for free tier; soft caps with Opus → Haiku downshift; nothing OSS tracks token spend

  • Goal entity + re-grounding on resume Versioned goal state; HITL-gated mutations; coherence check when workflow resumes after days
  • Custom annotation UI for domain experts Reviewer sees the artifact + 3-5 rubric questions, not spans/tokens. Build on Langfuse scores API

04The grand list of open holes

Twelve problems no OSS has shipped — and why each one matters.

The aggregated “Still unsolved” callouts from each gap section, plus a few that emerge only when you look across the whole stack. If you are picking projects, pick the ones that overlap with these — that's where engineering hours convert to product differentiation.

U01 · MEMORY

Cross-modal memory

Graphiti and mem0 are text-first. Meetings with embedded slides, threads with screenshots, docs with tables — no clean memory schema exists. Model multimodal nodes yourself.

U02 · WORKFLOWS

Goal coherence under indefinite suspension

A workflow paused 14 days for approval may resume into a world where the goal shifted, the data aged, the user's preferences changed. Re-grounding on every resume is DIY.

U03 · PROVENANCE

Multi-hop synthesis survival

Every popular framework treats citation as a property of the final answer, not something that propagates. Summarize a summary and they all break. This is the cleanest hole.

U04 · PROVENANCE

Inter-source contradiction handling

ConflictRAG and Micro-Act (2025-2026) identify retrieved-source conflict as the next frontier. No production OSS handles it. Build a detector on top of the lineage DAG.

U05 · DRAFTS

Semantic attribution under refactoring

Yjs 14's attributions tell you who inserted a character. When an agent rewrites a human paragraph, attribution either shatters per-character or collapses. Multi-agent conflict resolution on overlapping spans is also unhandled.

U06 · EVAL

Evaluating non-verifiable outputs

“Is this draft worth the user's time to edit” doesn't map to any generic eval axis. Build domain rubrics, validate against humans regularly, watch for judge drift.

U07 · EVAL

Counterfactual trajectory replay

Replay where changing the prompt changes which tools fire, which sub-agents spawn, which retrievals happen. Existing tools only replay single LLM calls, not whole agent trajectories.

U08 · DOCUMENTS

Charts and diagrams as data

Every parser extracts a chart bbox + caption; none reconstruct the underlying series reliably. Vision-LLM zero-shot sits at ~70% accuracy. Slide decks with composed graphics, financial filings with footnote chains, handwriting and whiteboards — all still need humans or paid commercial APIs.

U09 · TOOLS

OAuth scope-upgrade UX

User originally granted gmail.readonly; agent now needs gmail.send. No OSS handles the re-consent flow gracefully. Build it.

U10 · TOOLS

Per-tenant dynamic tool catalogs

Gateways give static allowlists. Tenant A wants Salesforce; B wants HubSpot. Per-tenant catalogs driven by actually-connected apps are DIY.

U11 · TOOLS

End-to-end cryptographic audit trail

Chat turn → tool call → upstream API with cryptographic linkage. The 2026 production-readiness bar. Nothing OSS ships it turnkey.

U12 · ENGINE

A clean primitive for “variants”

Every harness reinvents subagent dispatch. Handoffs vs subagents vs routers vs variants vs crews vs swarms — no agreed pattern. Write your own; expect it to be the most opinionated, fastest-changing piece.

The skeleton is on the shelf. The opinions are still the product.

In May 2026 the OSS landscape for knowledge-work agent infrastructure is the most complete it has ever been. Memory has finally been split into vector retrieval + bi-temporal graph + rerank, each layer with a clear best-in-class. Durable execution has matured to the point that single-binary engines like Restate ship awakeables as a first-class HITL primitive. Document understanding has closed the gap with commercial APIs on permissive licenses. Observability has coalesced around OpenTelemetry GenAI conventions. Citation grounding exists at the model layer. CRDT-based collaborative editing has the attribution primitive landing this year. MCP works and the Code Mode pattern solves the tool-explosion problem cleanly.

What hasn't changed: none of this assembles itself into a harness. The eight gap layers integrate through code you write — the queryLoop, the variant dispatch, the lineage DAG, the budget tracker, the goal entity, the OAuth scope-upgrade flow, the domain eval rubrics, the cross-hop ancestry UI. That code is the product. The five coding harnesses studied in this repo were right to write it themselves; a knowledge-work harness has even fewer reasons to outsource its core.

The deliverable from this report is simpler than it looks. Eight picks for the eight gaps. Three primitives worth borrowing without adopting the framework. Twelve open holes that overlap precisely with the places where engineering hours convert to defensible product. If you start tomorrow, you can be running the full stack against a prototype harness inside a sprint. The harder bet is whether the twelve open holes are the ones your users will actually care about — and that's a product question, not a tooling one.
