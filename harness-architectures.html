A reading of five open codebases

The shape of an agent is the shape of its loop.

Every coding agent has the same outer ring: ask a model, get tool calls, run them, feed results back. But beneath that, the five harnesses studied here resolve the same problem in strikingly different ways — different loop primitives, different permission models, different bets about where state lives. This is a side-by-side reading of pi, codex, Claude Code, code_puppy, and opencode, mapped to expose the architectural choices and the tradeoffs they imply.

pi
TypeScript · ~5K LOC core
Minimalist layered runtime — 4 npm packages composed via hooks. No native MCP, sub-agents, or permission popups; everything is an extension.
codex
Rust + thin TS shell
Safety-first. Tokio async loop, per-turn tool router, dual approval (rules + sandbox), Seatbelt/Landlock at the OS layer.
Claude Code
TypeScript · async generator
Generator-driven streaming. Yield points everywhere — tools, compaction, state mutations — so the UI sees every breath the agent takes.
code_puppy
Python · on Pydantic-AI
A thin shell on a fat framework. 50+ plugin hooks, two-pass tool registration, MCP-native, cascading model fallback.
opencode
TS / Bun + Go TUI
Client–server split (ACP). Loop runs client-side; sessions, tools, and history live server-side. Multi-device, offline-friendly by design.

02The loop

Every agent loop solves the same equation differently.

The job is identical: stream a model response, intercept tool calls, run them, feed results back, repeat until the model says it's done. The shape of how each harness organizes that — and what it makes visible to the outside — is the most consequential decision in the entire codebase.

Pick your loop primitive and you've picked your debugging surface, your cancellation story, and your concurrency model — all in one stroke.
pi
nested while · event-bus

Two stacked while loops. The inner handles a single turn — call model, run tools, repeat until the model emits no more tool calls. The outer picks up queued follow-up messages so the agent autonomously continues a conversation. Every state change goes out an event bus.

packages/agent/src/agent-loop.ts · runLoop()
codex
tokio channel · spawn per turn

An async-channel feeds submission_loop, which pattern-matches on operation kind and spawns a fresh run_turn task per user input. Each turn carries an immutable TurnContext — explicit, never thread-local — so concurrent turns never alias state.

codex-rs/core/src/session/handlers.rs · submission_loop
Claude Code
async generator · yield boundaries

One async function* with while (true) inside (query.ts:307). Every state transition — chunk arrived, tool ran, compaction kicked in, turn ended — exits the function via yield. The consumer sees the agent's lifecycle as a stream of events without polling.

src/query.ts · queryLoop (line 241)
code_puppy
thin wrap · retry decorator

The harness is mostly a wrapper. The actual loop is Pydantic-AI's; _runtime.py handles message-history prep, MCP server bring-up, and a @streaming_retry that catches transient stream errors. The plugin hook system runs above and around the call.

code_puppy/agents/_runtime.py · run_with_mcp
opencode
client/server · ACP events

The client creates a message, then waits on an event stream. Tool calls arrive as permission.asked; the client decides yes/no and sends approval back. The actual model call, tool execution, and message persistence all live server-side — the client is mostly UI and consent.

03Tool dispatch

Five answers to "which tools, when, in parallel?"

Tool registration looks like a config detail until you realize it gates safety, performance, and extensibility all at once. These five harnesses split into static-list, partition-by-safety, per-turn-router, two-pass, and server-side dispatch — and each tradeoff shows up in the developer experience downstream.

Static list, mode-tagged
pi

Tools register once at boot with an executionMode. The loop calls Promise.all by default and falls back to a sequential for…await for tools that mark themselves serial. Permission gates are hooks, not policy types.

// before/after hooks wrap every call
{ executionMode: 'parallel' } → Promise.all
{ executionMode: 'sequential' } → for…await
beforeToolCall?(call) → may reject → tool errors
afterToolCall?(call, result) → may transform
schema: JSON Schema parallel: by-default extension: hook
Per-turn ToolRouter rebuild
codex

Every turn re-queries MCP servers, re-evaluates plugin eligibility against the current config, and constructs a fresh ToolRouter from MCP tools + plugins + accessible connectors. Tools that need OAuth sit in a deferred bucket.

ToolRouter::from_turn_context(
  &turn,
  mcp_tools,         // dynamic
  deferred_mcp_tools, // pending auth
  discoverable_tools,
  extension_tool_executors,
  dynamic_tools,
)
schema: JSON Schema (MCP) parallel: FuturesOrdered rebuild: every turn
Partition by concurrency-safety
Claude Code

Each tool declares isConcurrencySafe(input). The dispatcher walks the LLM's tool calls and greedily batches consecutive safe ones; anything write-y breaks the batch and runs serially. Cap is 10 parallel (env-overridable).

// reduce into batches by safety
if (safe && lastBatch.safe) lastBatch.push(t);
else newBatch({safe, [t]});

// safe batch → all(), unsafe → sequential
schema: Zod parallel: hybrid cap: 10
Two-pass registration
code_puppy

Step one: build a dummy Pydantic-AI agent to enumerate the names of native tools. Step two: load MCP servers, drop any whose names collide, and rebuild for real. Solves the MCP name-clash problem without manual config.

dummy = PydanticAgent(model, toolsets=[])
existing = {t.name for t in dummy._tools}
mcp = load_mcp_servers(name)
mcp = filter_conflicting(mcp, existing)
agent = PydanticAgent(model,
  toolsets=[*native, *mcp])
schema: Pydantic parallel: Pydantic-AI MCP-first
Server-side dispatch
opencode

The client never invokes a tool. It receives a permission.asked event, decides, and posts approveTool/ignoreTool. The server runs every tool, persists every result, and streams deltas back — so tools can be slow, async, or remote without freezing the UI.

// client
events.on('permission.asked', async e => {
  const ok = await permissions.request(e);
  sdk.message[ok ? 'approveTool' : 'ignoreTool'](e);
});
// server: pending → running → completed/error
schema: server-defined parallel: server-orchestrated state machine: 5-state
04Context & compaction

When the window fills, who decides what to forget?

Context-window management is the unglamorous core of an agent harness. Get it wrong and the agent forgets the task at hand; get it right and it survives day-long sessions. Each harness has a different theory of when to compact, how aggressively, and how visible the operation is to the user.

LLM-summarized, branch-preserving
pi

When tokens cross a threshold, the harness summarizes the oldest N messages via the LLM and writes a special "compaction" record to the JSONL log. The original messages are not deleted — they remain in the file, so a user can branch off the pre-compaction state.

Implication: full audit trail. Disk grows; active context doesn't.

Three-phase auto-compact
codex

Compaction fires at three trigger points: pre-sampling (before the LLM call), mid-turn (when tokens exhaust during a streaming turn), and post-sampling (cleanup). The AutoCompactWindow distinguishes ServerObserved prefill (authoritative) from Estimated (client guess).

Implication: bridges streaming latency without over- or under-counting tokens.

Four strategies, all observable
Claude Code

Layered defense: autocompact summarizes old turns, microcompact shaves on output-token exhaustion, snip trims oversized tool outputs, and context collapse is the nuclear fallback. Every compaction step yields a CompactProgressEvent — the user watches the prompt being engineered.

Implication: maximum transparency, maximum complexity to debug.

Summary + protected window
code_puppy

A summarize() function that preserves system messages, the most recent exchange (protected token count), and turns everything in between into a summary. Plugins can replace the strategy via history_processors.

Implication: simple to reason about; plugin-replaceable when simple isn't enough.

A variant agent does the work
opencode

Compaction isn't a function — it's its own agent variant. When the threshold trips, the parent agent calls getAgentVariant('compaction').prompt(...), treating summarization as a first-class delegation. Old messages are replaced with the summary on the server.

Implication: same primitives all the way down. The cost of summarization shows up in your normal token-usage accounting.

05Permissions & sandboxing

From "trust the user" to "trust the kernel."

Permission models reveal what each project is afraid of. pi and code_puppy assume you'll add policy via plugins. Claude Code and opencode build in modes and approval flows. Codex pushes furthest — into OS-level sandboxing — because it was designed to run untrusted commands without human-in-the-loop.

Dual approval + OS sandbox
codex

The AskForApproval enum has a Granular variant that separates rules approval (allow/deny lists) from sandbox approval (OS-level confinement). A command can pass rules but still be confined; or run without a sandbox but only if explicitly allowlisted.

OS sandboxing is plumbed by injecting env vars (CODEX_SANDBOX_ENV_VAR="seatbelt") — the runtime reads them and applies Seatbelt on macOS, Landlock+seccomp on Linux. Decoupling policy selection from enforcement is a deliberate choice.

Four named modes
Claude Code

Tools declare a ToolPermissionMode ∈ {default, acceptEdits, bypassPermissions, plan}. Before any tool runs, a canUseTool(name, input) callback fires — host apps decide whether to prompt the user, auto-approve, or deny.

Plan mode is structurally distinct: the agent describes what it would do rather than doing it. Useful for high-stakes review before committing to writes.

Three-rule merge
opencode

Per tool, the effective rule is one of once / always / reject. once caches the user's first answer for the rest of the session. The actual rule is a 3-way merge: global defaults ← variant defaults ← user overrides.

Sub-agents run in subagent mode, which silently strips write tools — defense in depth.

Hook-defined policy
pi

No built-in permission system. The beforeToolCall hook receives each invocation; the hook can validate, reject, or transform args. Whether to prompt the user, log, or sandbox is up to the application embedding the harness.

Tradeoff: maximum flexibility, zero defaults. Easy to ship something dangerous.

50+ hooks, any policy
code_puppy

Plugins implement callbacks like file_permission(agent, path, op) → bool, run_shell_command(agent, cmd) → bool, pre_tool_call(agent, name, args) → bool. Each plugin can add async approval flows by deferring to external services.

MCP servers run in their own processes — process isolation is free, kernel sandbox is not.

06Sub-agents

Delegate how? Share what?

The decision to support sub-agents is a decision about state. Do they share context (cheap, leaky) or get a fresh slate (clean, but you pay tokens to re-onboard them)? Do they inherit your permissions (convenient, scary) or run constrained (safe, awkward)?

Not native, by design
pi

The pi core has no sub-agent primitive. If you want delegation, you wire it as a tool whose handler spins up its own agent loop. The framework's bet: opinionated sub-agent semantics live above the core, not in it.

Registry · spawn slots · policy inheritance
codex

An AgentRegistry per session tracks every spawned agent, enforces a depth limit (slot reservation), inherits the parent's approval+sandbox policies, and gives each agent a human-readable nickname for logs.

Inter-agent communication runs through the session's event channel — the same mechanism users use.

Fresh agent via Task tool
Claude Code

The Task tool spawns a brand-new agent with isolated context. The parent passes a goal; the child runs to completion, cancellation, or timeout; and the final output is returned as a tool result. The child can be scoped to a subdirectory and a sandbox mode.

Cost: context isolation means the child re-reads what it needs. Benefit: parent context doesn't bloat with exploratory tool output.

Subclass & identity
code_puppy

Each concrete agent is a subclass of BaseAgent with its own system prompt, tool set, and message history. A unique identity string (name-{id[:6]}) lets agents reference one another for coordination.

Per-agent history prevents cross-leakage but means coordination is by tool call, not shared memory.

Variant + shared session
opencode

Sub-agents are just variants (plan, explore, scout, …) running in subagent mode with read-only tool access. They share the parent's session — so their outputs append to the same message stream and inherit the same prompt cache.

Compaction itself is implemented as a sub-agent variant.

07Providers & persistence

Where does the model live, where does the state go?

Two adjacent choices that shape an agent's deployment story. Provider abstraction decides what you can swap; persistence decides what survives a crash. Together they determine whether the agent is "a CLI you run" or "a service your team operates."

pi-ai: native multi-provider
pi

A dedicated pi-ai package normalizes Anthropic / OpenAI / Google / Azure / DeepSeek / Mistral / others behind a single LMFunction type that yields token strings. The Anthropic adapter takes advantage of prompt caching via cache_control; other adapters do what they can.

Persistence: JSONL on disk in ~/.pi/agent/sessions/, append-only with a parentId field that turns the file into a conversation tree.

ModelProvider trait + rollout log
codex

A ModelProvider trait exposes capabilities (e.g., supports_remote_compaction) so the harness can adapt strategy per provider. Transports degrade gracefully: WebSocket primary, HTTPS fallback after N retries.

Persistence: a rollout event log. Resume scans backward until it finds a compaction checkpoint, then replays forward — efficient even for very long sessions.

Anthropic-first, host-pluggable
Claude Code

The harness clearly assumes Anthropic semantics (system prompts, cache_control, streaming format), but the sendMessage shim is generic enough that host apps can wire alternative providers.

Persistence: a JSON file per session in ~/.claude/projects/<project-id>/sessions/, plus user-authored CLAUDE.md memory files discovered via directory walk on startup.

Pydantic-AI + cascade fallback
code_puppy

Pydantic-AI's provider integrations do the heavy lifting; on top, ModelFactory + load_model_with_fallback implement a cascade — if your preferred model errors out, try the next, then the next.

Persistence: minimal. Message history lives in BaseAgent._message_history with a _compacted_message_hashes set to avoid re-summarizing the same content. Durability is a plugin's job.

SDK-mediated, switch mid-session
opencode

Models are addressed by string; the SDK routes to the right provider. The interesting part: the conversation is server-state, so you can swap models mid-session and keep the entire history intact.

Persistence: server-side database, accessed by clients via SDK. Multi-device resume and offline-first patterns are first-class.

08Implications

What you pick for, what you give up.

Each architectural choice is a bet about what matters most in the next ten thousand lines of code. Here are the bets these five harnesses are making — and the consequences they accept.

The loop primitive is the harness's debugging surface.

cross-cutting
  • Generators (Claude Code) make every state transition observable from the consumer — but stack traces become harder, and partial state across yields is a category of bug.
  • Async channels (codex) are great for backpressure and explicit cancellation — but the message types proliferate, and the loop body becomes a giant match.
  • Nested whiles (pi) are the most legible — but the event bus is your tracing tool, not the loop itself.
  • Event subscriptions (opencode) separate UI thread from work thread cleanly — but retry semantics, ordering, and reconnection are now problems you own.
  • Framework wrapper (code_puppy) is fast to ship — but you inherit the framework's loop semantics, and customization stops where Pydantic-AI's primitives stop.

Hook-based permissions trade defaults for flexibility.

safety

pi and code_puppy push permission policy entirely into the application layer. The win is that any policy — user prompts, allowlists, external approval services, audit logging — composes naturally. The cost is that the default behavior is permissive: forget to wire a beforeToolCall hook and your agent will happily rm -rf.

Claude Code and opencode codify a small set of modes/rules; codex goes furthest and codifies OS-level enforcement. The more the harness chooses for you, the safer the default — and the harder it is to express unusual policies (e.g., "ask only when this regex matches").

Client/server is a deployment story, not a feature.

opencode

Opencode's biggest architectural choice — running the agent loop client-side but state and tools server-side — is what makes multi-device resume, offline-first, and slow tools possible. None of those come for free; they require a server, an SDK contract, and an event subscription protocol.

Other harnesses can simulate parts of this (codex's rollouts let you resume; pi's JSONL lets you replay) but a true client/server split is a different category of product. You ship a service, not a binary.

Sandboxing is a one-way door.

codex

Codex's Seatbelt/Landlock integration unlocks "full-auto" modes that the other harnesses cannot safely offer. The investment is real: profile authoring, per-OS implementation, env-var conventions, and a granular approval enum that distinguishes rules from sandbox enforcement.

For agents that need to run untrusted commands without human approval — CI, batch refactors, headless scripts — there is no substitute. For agents that have a human in the loop, the work may not pay for itself.

"Compaction is normal" vs "compaction is a button."

context

Two postures exist. Hidden compaction (pi, code_puppy) treats it as an internal optimization — fires when needed, doesn't make a fuss. Observable compaction (Claude Code, opencode) treats it as a first-class user-visible event — Claude Code yields progress; opencode literally runs a separate agent to do it.

Visible compaction wins when users care about why the agent forgot something, because they can see the moment and the reason. Hidden compaction wins when the agent is mostly autonomous and the answer to "what got compacted?" is "irrelevant."

Sub-agent context isolation is a tax — and an investment.

delegation

Claude Code's Task tool spawns a fresh agent with no parent context. The cost is token waste — the child re-reads what it needs. The benefit is enormous: the parent context stays small, and the child can explore a problem aggressively (read 50 files, run 100 greps) without polluting the main conversation. This is why Task is the only realistic way to research a large codebase without context death.

Opencode's shared-session sub-agent is the opposite bet — token-efficient, but exploration leaks into the main thread. The right call depends on whether your sub-agent's job is a delegated sub-problem or a coordinated workflow step.

"Minimal core" is itself a bet.

pi · code_puppy

Pi and code_puppy both lean into "thin core, extend everything." Pi defers MCP, sub-agents, plan mode, and permissions to extensions. Code_puppy defers… nearly everything to its 50+ plugin hooks.

The bet is that ecosystems form. If they do, the core stays small and clean. If they don't, you end up writing every plugin yourself, and the "minimal core" becomes a TODO list rather than a feature. The biggest risk of this design is also its appeal: you have to build the rest.

The opinionated parts are where the value is.

conclusion

The shared scaffolding — loop, tool dispatch, message history — is converging. Read all five and you will recognize the same skeleton everywhere. What differentiates these projects is the small set of opinionated choices stacked on top: Claude Code's partition-by-safety, codex's dual-approval + sandbox, opencode's client/server split, pi's hook-everywhere ethos, code_puppy's plugin proliferation.

If you are building a harness, the skeleton is table stakes. The opinions are the product. Decide what you believe about safety, observability, delegation, and persistence before you write the loop — because the loop will encode those beliefs whether you intend it to or not.

pi

TypeScript · earendil-works/pi · packages/coding-agent + agent + ai + tui

Philosophy

Minimalist composable runtime. The codebase is split across four npm packages so each is independently useful: pi-ai is just an LLM client; pi-agent-core is just a loop; pi-tui is just rendering; pi-coding-agent ties them together. The agent core is ~2K LOC. Everything beyond — MCP, sub-agents, plan mode, permission popups — is an extension.

Loop

Two stacked while loops in agent-loop.ts. The inner loop drives a single turn; the outer loop processes follow-up messages queued after the agent terminates. Steering (user input mid-turn) is detected at iteration boundaries.

while (followUpMessages.length > 0 || !terminated) {
  while (steeringMessages.length > 0 || !turnEnded) {
    const response = await lmFunc(...);  // streaming
    if (toolCalls) await executeToolCalls(...);
    // emit events: turn_start, message_*, tool_execution_*
  }
}

Tools

Static list. Each tool can declare executionMode: 'parallel' | 'sequential'. Default is parallel via Promise.all. Hooks (beforeToolCall, afterToolCall) gate and transform invocations.

Context

JSONL log per session with a parentId field, giving native conversation branching. Compaction summarizes old turns via the LLM and writes a compaction record to the log — but original messages stay in the file for replay.

Permissions

No built-in policy. The beforeToolCall hook is the entire permission API.

Sub-agents

None native. Build them as tools whose handlers spawn a child loop.

Standout

Aggressive separation of concerns. The event-driven core is unusually clean — a future test harness or alternative UI just subscribes to the same event types the TUI does.

codex

Rust · openai/codex · codex-rs/core/src/session/*

Philosophy

Safety-first. Rust for memory safety and crisp concurrency primitives; OS-level sandboxing for untrusted command execution; explicit context passing instead of thread-locals. The architecture pays the up-front cost of more types and more files in exchange for an agent you can safely automate.

Loop

A tokio task running submission_loop pulls Submission messages off an async_channel and pattern-matches the operation. UserInput spawns a fresh run_turn task carrying an immutable Arc<TurnContext>. Multiple turns can run concurrently without aliasing state.

Tools

The ToolRouter is rebuilt every turn from MCP tools (queried live), plugins (re-evaluated against current config), and accessible connectors. Tools awaiting OAuth sit in a deferred bucket. FuturesOrdered + tokio::select! multiplex tool execution against the streaming response.

Context

Three compaction triggers: pre-sampling, mid-turn (token exhaustion during a stream), and post-sampling. The AutoCompactWindow distinguishes ServerObserved prefill (authoritative) from Estimated (client guess) so the budget is accurate even during streaming.

Permissions

The AskForApproval::Granular { allows_rules_approval, allows_sandbox_approval } variant separates allowlist policy from OS confinement. A command can be allow-listed but still confined; or sandbox-exempt but approval-gated.

Sandboxing

Seatbelt on macOS, Landlock + seccomp on Linux. Selection happens via env-var injection at spawn — CODEX_SANDBOX_ENV_VAR="seatbelt" tells the sandbox runtime to apply the appropriate profile. Network-disabled mode is its own env var. Decoupling policy from enforcement keeps the sandbox runtime independent.

Sub-agents

An AgentRegistry per session tracks every spawned child. Slot reservation enforces depth limits; policy inherits from parent; each agent gets a human-readable nickname for logs.

Persistence

Rollouts are append-only event logs. Reconstruction scans backward from the end of the log to find the most recent compaction checkpoint, then replays only events after it — O(recent activity), not O(history).

Claude Code

TypeScript · reverse-engineered · src/query.ts

Philosophy

Streaming-first. The loop is an async generator so the consumer (UI, test, telemetry) sees every state transition as an event without polling. Every long-running operation — tool calls, compaction, even error recovery — happens at a yield boundary, which means cancellation is just "stop iterating."

Loop

async function* queryLoop() at query.ts:241 with while (true) at line 307. State (messages, tool context, compaction tracking, turn count) is destructured at the top of each iteration, mutated, re-bundled, and re-entered. Every interesting event is yielded.

async function* queryLoop(params, consumedUuids) {
  // mutable State { messages, toolUseContext, autoCompactTracking, ... }
  while (true) {
    for await (const ev of sendMessage(...)) yield ev;
    if (inputTokens > threshold) yield* compact(state);
    if (toolUseBlocks.length) yield* runTools(state);
    if (response.stop_reason === "end_turn") return Terminal{...};
  }
}

Tools

Static list, gated by a Bun feature() macro and USER_TYPE env. Each tool exposes isConcurrencySafe(input); the dispatcher batches consecutive safe tools into parallel groups (capped at 10) and runs anything write-y serially. Schemas are Zod.

// partition by safety, max 10 parallel
toolUses.reduce((batches, t) => {
  const safe = isConcurrencySafe(t);
  if (safe && last.safe) last.push(t); else newBatch(safe, t);
});

Context

Four named compaction strategies — autocompact (LLM summary of old turns), microcompact (output-token exhaustion recovery), snip (trim oversized tool results), collapse (last resort). Every step yields a CompactProgressEvent so the user sees the prompt being engineered.

Permissions

Tools declare a ToolPermissionMode ∈ {default, acceptEdits, bypassPermissions, plan}. A canUseTool(name, input) callback runs before every invocation; the host decides whether to prompt, auto-accept, or deny. Plan mode is structurally distinct — the agent describes intent without executing.

Sub-agents

The Task tool spawns a fresh agent with isolated context, optionally restricted to a subdirectory or sandbox mode. The child's final output is the tool result; intermediate state never bleeds into the parent.

Persistence

A JSON file per session in ~/.claude/projects/<project-id>/sessions/. Plus per-project CLAUDE.md memory files discovered via directory walk on init.

code_puppy

Python · mpfaffenberger/code_puppy · built on Pydantic-AI

Philosophy

Thin shell on a thick framework. Pydantic-AI provides the loop and tool primitives; code_puppy adds plugin infrastructure, MCP integration, retry semantics, and a TUI. The golden rule (in AGENTS.md): extend via plugins, not by modifying core.

Loop

run_with_mcp() in _runtime.py orchestrates: prune interrupted tool calls from history, load MCP servers, build the Pydantic-AI agent, invoke run_stream, retry on transient errors. The retry decorator is the most distinctive piece — it matches on error substrings (rate_limit, overloaded, RemoteProtocolError, ReadTimeout) and re-enters streaming without losing position.

Tools

Two-pass registration to handle MCP name collisions. First pass: build a dummy Pydantic-AI agent with no toolsets and inspect _tools for native names. Second pass: load MCP servers, drop colliders, and rebuild the real agent.

dummy = PydanticAgent(model, toolsets=[])
existing = {t.name for t in dummy._tools}
mcp = filter_conflicting(load_mcp_servers(name), existing)
real = PydanticAgent(model, toolsets=[*native, *mcp])

Context

An _history.py module provides token estimation, hash-based dedup, and orphan pruning — removing incomplete tool-call sequences so a re-entered conversation doesn't trip on malformed history. _compaction.py summarizes with a protected token count for system + recent messages.

Permissions

50+ plugin hooks defined in callbacks.py — startup, shutdown, invoke_agent, load_prompt, run_shell_command, file_permission, pre_tool_call, post_tool_call, stream_event, and many more. Permissions are implemented as plugins, not as harness primitives.

Sub-agents

Subclassing BaseAgent. Each instance gets a unique identity ({name}-{id[:6]}) for coordination. History is per-agent — no shared mutable state.

Providers

Per-agent model pinning falls back to a global default, then cascades through a configured list. Robust to API outages at the cost of mid-conversation model swaps changing the agent's behavior subtly.

opencode

TypeScript (Bun) + Go TUI · sst/opencode · packages/opencode

Philosophy

Client/server by design. The agent loop runs on the client (so it can be a CLI, a TUI, a web UI, a Slack adapter, …) but session state, message history, and tool execution live server-side. The Agent Control Protocol (ACP) is the contract between them.

Loop

The client's prompt() creates a message via SDK, then awaits a message.part.completed event. In between, permission.asked events arrive for tool calls — the client decides yes/no and replies with approveTool/ignoreTool. The actual work happens in session/processor.ts server-side.

// client side
async prompt(input) {
  const msg = await sdk.message.create({sessionId, input});
  return new Promise(resolve => {
    events.subscribe(e => {
      if (e.type === 'message.part.completed' && e.id === msg.id) {
        resolve(e.data);
      }
    });
  });
}

Tools

Server-side dispatch. Per tool call, the server emits permission.asked; the client responds. Tools transition through a 5-state machine: pending → running → completed | error | ignored. Long-running tools stream deltas via message.part.delta.

Context

Compaction is its own agent variant. When the threshold trips, the harness invokes getAgentVariant('compaction').prompt(history) and replaces old messages with the result via sdk.message.replace. Same primitives all the way down.

Permissions

Three rules per tool: once, always, reject. Effective rule comes from a three-way merge: global defaults ← variant defaults ← user overrides. Sub-agents run in subagent mode, which silently strips write tools.

Sub-agents

Variants: build, plan, explore, scout, general, compaction. Sub-agents share the parent session — they see and append to the same message stream — but are constrained by mode-restricted tool access.

Providers

Multi-provider via the SDK. The killer feature: you can switch models mid-session and keep the full history intact, because conversation state is server-state.
