Design report — 2026 · grounded in five coding harnesses

What a knowledge-work harness needs that a coding one doesn’t.

Coding harnesses — pi, Claude Code, codex, code_puppy, opencode — are the closest thing the field has to reference architectures for agents. They give us a vocabulary, not a blueprint. Knowledge work shifts the architectural pressures in three specific ways, and naively porting one of these harnesses to research, drafting, scheduling, or triage quietly degrades safety, observability, and recall. This report names the differences, proposes six decisions, and identifies four problems no coding harness has had to solve.

The architectural commitments
01Three axes of difference

Knowledge work changes what the harness has to model.

Coding agents are anchored to a compiler, write to a recoverable filesystem, and treat the repo as ground truth. Knowledge work has none of those affordances — and the harness has to compensate for each one architecturally.

AXIS · 1

Verification asymmetry

The coding loop is anchored to fast, brutal ground truth — tsc, pytest, lint, the next tool call. Claude Code’s queryLoop can emit edits with confidence because the environment will answer back in seconds.

Knowledge work has no equivalent. There is no test runner for "draft a board update." The "did it work" signal has to be modeled into the loop itself — critique sub-agents, draft/refine cycles, human checkpoints. Tool results are not self-validating.

Coding tsc · pytest · lint
next tool call disambiguates
Knowledge work critique variant · user checkpoints · draft cycles
AXIS · 2

Blast radius asymmetry

Coding harnesses assume the worst case is "modify a file on disk." git reset recovers most damage. Permission models like Codex’s AskForApproval are tuned for friction-minimization on a population of mostly-recoverable actions.

Knowledge work has a different action distribution. A small minority of tool calls — send_email, create_event, share_doc, post_slack — are not undoable. The permission model has to recognize this asymmetry, not treat all writes as equivalent.

Coding rm -rf → annoying
git push -f → recoverable
Knowledge work wrong-recipient email → permanent
triple-booked CEO → expensive
AXIS · 3

Source-of-truth asymmetry

A coding agent has the filesystem and git history. These are complete — anything not in them definitionally isn’t the project. Claude Code injects git status confidently because git is authoritative.

Knowledge-work context is scattered, lossy, conversational. Facts live in Slack threads, draft Notion docs, unread emails, prior conversations. The conversation itself is often the artifact, not a scratch surface over an authoritative substrate. Memory strategy has to do real architectural work.

Coding filesystem + git = ground truth
memory = scratchpad
Knowledge work no ground truth
conversation = artifact
02Six architectural decisions

From the differences, six load-bearing choices.

Each decision below cites which of the five harnesses it borrows from, with the specific file path or pattern. Three of them are reshape moves; three are direct lifts; one — context strategy — is where the inheritance breaks down hardest.

DECISION · 01

Loop shape: event-driven, async-generator, steerable

Borrow Claude Code’s yield-everything queryLoop and layer pi’s steering/follow-up split on top.
Borrowed from Claude Code pi

Coding feels fast because tool calls return in milliseconds. Knowledge work is opaque — a 30-minute deep-research turn gives the user nothing to look at. The yield boundary in Claude Code’s queryLoop is the only place to surface machinery: sub-agent progress, compaction events, citation grounding. Showing the work matters more when verification is slow.

Pi’s distinction between steering messages (interrupt mid-turn) and follow-up messages (continue after turn) becomes load-bearing. Hours-long research threads cannot be restarted because the user said "also check the 2023 numbers." 

async function* knowledgeLoop(state) {
  while (state.followUps.length || !state.terminated) {       // pi outer
    while (state.steering.length || !state.turnEnded) {       // pi inner
      yield* runTurn(state);                                  // Claude Code yields
      if (state.tokensExceeded) yield* compact(state);        // observable compaction
      if (state.steering.length) state = applySteering(state);
    }
    state = await consumeFollowUp(state);
  }
  return state.result;
}

DECISION · 02

Context strategy: layered memory, not just compaction

This is where coding-harness inheritance breaks down hardest. None of the five do this well.
Borrowed from Claude Code pi
Net-new Persistent semantic memory tier

All five harnesses treat memory the same way — a sliding window, compacted when full, plus a static CLAUDE.md/AGENTS.md file. This works for coding because the repo is ground truth; the agent can always re-read a file. Conversation is a scratchpad over an authoritative substrate.

Knowledge work has no such substrate. The conversation is the substrate. Compaction is lossy compression of the only copy. So:

TierPurposeInspiration
Working context Current turn, full fidelity Claude Code autocompact/microcompact/snip
Session memory Append-only log, branchable pi JSONL with parentId fork
Persistent memory Cross-session semantic retrieval, provenance metadata Not present in any of the five

The third tier is the gap. None of the five harnesses do retrieval well — they lean on grep and read, which is adequate for code. Knowledge work demands an embedding store, retrieval at turn start, and provenance metadata on every fragment so the agent can cite where a recalled fact came from.

src/query.ts compaction cascade (Claude Code) · ~/.pi/agent/sessions/{id}.jsonl (pi)
DECISION · 03

Permissions: granular and irreversibility-aware

Borrow Codex’s two-axis shape; reframe axes for blast radius; layer opencode’s three-tier UX.
Borrowed from codex opencode Claude Code

Codex’s GranularApprovalConfig in exec_policy.rs is the right shape — separating rules approval from sandbox approval — but the axes are coding-flavored. For knowledge work, the axes are blast radius and reversibility:

R0 · Read

Search · Fetch · Query

search_inbox, get_doc, query_crm

Auto-allow · silent
R1 · Draft

Local write

create_draft, save_scratch_note

Auto-accept · toast
R2 · Send

External, irreversible

send_email, post_slack, share_doc, create_event

Always ask · preview required
R3 · Destroy

Destructive

delete_record, cancel_event, archive_thread

Always ask · diff + rollback path

The critical move is that preview becomes part of the permission UX, not an afterthought. Claude Code’s "plan" mode in src/Tool.ts already prefigures this — for knowledge work, plan mode becomes the default for any external-send action. Opencode’s once / always / reject tier handles the repetition problem: a user approving "send email to Bob" should not re-approve forty times in the same session.

DECISION · 04

Tool surface: MCP-first, two-pass registration

Borrow code_puppy’s probe-then-filter; layer codex’s per-turn re-registration.
Borrowed from code_puppy codex

Knowledge-work tools live almost entirely in SaaS — Gmail, Notion, Linear, Drive, Calendar, Slack, Airtable. MCP has already become the lingua franca. Two patterns from the harnesses transfer cleanly:

code_puppy’s two-pass registration probes for native tool names with an empty toolset first, then loads MCP servers and filters conflicting names before rebuilding. Without this, two MCP servers each exposing a search tool silently shadow each other — the agent uses the wrong one with no obvious failure. 

# Pass 1: probe
dummy = PydanticAgent(model, toolsets=[])
existing = {t.name for t in dummy._tools}

# Pass 2: filter and rebuild
mcp_servers = filter_conflicting_mcp_tools(load_mcp_servers(), existing)
agent = PydanticAgent(model, toolsets=[*native, *mcp_servers])

codex’s per-turn registration rebuilds the ToolRouter every turn from MCP inventory + plugins + connectors. A user may connect Gmail mid-session, OAuth scopes may expire, a workspace may be added — the tool set is not stable, and re-registering each turn absorbs that volatility cleanly (cached for the duration of a turn).

code_puppy/agents/_builder.py two-pass registration · codex turn.rs built_tools() per-turn
DECISION · 05

Sub-agents: variants over forks

Borrow opencode’s variant model; reserve Claude Code Task-tool isolation for parallel research.
Borrowed from opencode Claude Code

Opencode’s AgentVariants table (build, plan, explore, compaction) maps almost perfectly onto knowledge work. The corresponding variants: 

export const KnowledgeVariants = {
  primary:   { name: 'primary',   role: 'user-facing assistant' },
  research:  { name: 'research',  role: 'gather and summarize source material' },
  draft:     { name: 'draft',     role: 'compose deliverables (emails, docs, plans)' },
  critique:  { name: 'critique',  role: 'review drafts for accuracy, tone, citations' },
  summarize: { name: 'summarize', role: 'compress conversation for memory' },
}

Variants share the session and message history (read-only for sub-agents) but get distinct system prompts and permission rules. The critique variant gets read-only access and a prompt that hunts for unsupported claims, tone mismatches, missing citations — substituting for the compiler in the verification asymmetry.

Reserve Claude Code’s Task-tool pattern for when you specifically want noise kept out of the parent context — a research task about to read fifty web pages shouldn’t bloat the main conversation. The Task tool returns only the synthesis.

DECISION · 06

Persistence: append-only JSONL + derived artifacts

Borrow pi’s substrate; add a layer nobody has — the user-facing unit is artifacts, not turns.
Borrowed from pi codex
Net-new Derived artifacts index over messages

Pi’s session format — JSONL at ~/.pi/agent/sessions/{id}.jsonl with optional parentId for branching — is the right substrate. Human-readable, append-only, version-control-friendly, trivially forkable.

But messages are not the user-facing unit of knowledge work. The user does not care about turn 47 of conversation X. They care about the document that was drafted, the decision made, the email sent, the meeting scheduled. Maintain a derived artifacts view alongside the JSONL:

ArtifactContentProvenance
DraftDocument / email bodyTurns producing it · sources cited
DecisionStatement of intentConversation · ratifying turn N
Sent actionEmail · post · eventPermission grant ID · sent-from turn
SynthesisResearch outputSource list with retrieval scores

Codex’s reverse-replay reconstruction pattern in rollout — scan backward to the most recent compaction checkpoint, then apply selective forward events — is the efficient way to rebuild the artifacts view on session resume.

04Where the inheritance ends

Four problems none of the five solve.

A knowledge-work harness that stops at "merge the five coding harnesses’ best ideas" is incomplete. These four gaps are not solved in any of the studied codebases because coding-agent authors haven’t had to face them. This is where the real engineering investment goes.

GAP · 01

Retrieval-augmented memory across sessions

The five harnesses lean on filesystem reads (grep, find, Read) as a memory substitute — fine when the repo is ground truth, useless when the substrate is conversational.

  • Embedding store over every artifact, decision, draft, turn-level snippet
  • Retrieval triggered automatically at turn start, conditioned on the question
  • Provenance metadata: session_id · source · timestamp · author · contradicted-by
  • Staleness signals — a six-month-old "deadline" memory cannot be trusted blindly
→ Closest analog: CLAUDE.md (static and global, no retrieval)
GAP · 02

Proactive, long-horizon loops

All five harnesses are reactive — user prompts, agent runs, agent stops. Knowledge work often needs proactive behavior:

  • "Watch this customer’s tickets for a month and flag churn risk"
  • "Every Monday at 9am, summarize PRs merged last week to the eng channel"
  • "If the SEC files anything about <competitor>, ping me within an hour"

Requires scheduler primitives, self-updating goals, budget ceilings, idempotency so a watcher doesn’t re-send the same alert each wake.

→ Closest analog: codex submission_loop (right shape, no scheduling primitive)
GAP · 03

Source provenance, surfaced in the UI

When a knowledge agent claims "the Q3 deadline is May 15," the user needs to know where that came from. Coding agents don’t need this because the code is self-evidencing; a knowledge agent’s claims are synthesized and have no inherent verifiability.

  • Every retrieved fact carries (source_id, url, retrieved_at, confidence)
  • LLM emits inline citations like [src:gmail-thread-abc]
  • UI renders citations as clickable, hoverable, showing source snippet
  • Synthesis preserves lineage: "X is true because S₁ and S₂"
→ Closest analog: none. Well-trodden in academic LLM literature, absent from harness architecture.
GAP · 04

Draft / refine cycle as a first-class state

Message turns are the wrong granularity. "Draft an email, get my approval, revise based on edits, send" is not three turns — it is one logical operation in three phases, and the harness should model that explicitly.

  • A Draft is a first-class persisted entity, separate from messages
  • Drafts have versions; each version is a delta from the prior
  • User feedback (inline edits, comments, ratings) is structured input routed back to the draft variant
  • "Send" is bound to a specific draft version — ratifying a different version re-confirms
→ Closest analog: opencode tool state machine (operates on tool calls, not artifacts)
05Putting it together

A minimum-viable knowledge-work harness.

The architectural sketch: each layer borrows its primitive from the harness color-coded next to it. The amber blocks are the four gaps — the parts no coding harness ships.

Claude Code pi codex code_puppy opencode ✦ new — gap to build

The skeleton is table stakes. The opinions — and the four gap items — are the product.

The five coding harnesses are not a roadmap. They are a vocabulary. Each contributes one primitive worth keeping; none was built with the right pressures in mind to be a complete answer.

A knowledge-work harness that takes loop shape from Claude Code, persistence from pi, permission granularity from codex, agent variants from opencode, and MCP integration from code_puppy gets the skeleton right for free.

What remains — and what the field has not yet shipped — is memory that spans sessions, proactivity that survives the user logging off, citations that survive synthesis, and drafts that survive revision, all as first-class objects.

Those four are not features. They are the architectural commitments that decide whether the agent is a chat interface or a colleague.
