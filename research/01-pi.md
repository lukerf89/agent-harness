# Pi Agent Harness: Architectural Analysis

## 1. Overview

The Pi agent harness is a minimal, self-extensible TypeScript-based agent runtime designed to power interactive coding agents. Unlike monolithic frameworks, Pi follows a layered, composable architecture where core concerns are separated into four npm packages:

- **@earendil-works/pi-ai** (packages/ai): Multi-provider LLM abstraction supporting Anthropic, OpenAI, Google, Azure, DeepSeek, Mistral, and others
- **@earendil-works/pi-agent-core** (packages/agent): The event-driven agent loop, state machine, and session management harness
- **@earendil-works/pi-tui** (packages/tui): Terminal UI components with differential rendering
- **@earendil-works/pi-coding-agent** (packages/coding-agent): Interactive CLI that ties everything together

The philosophy is radical minimalism in core with aggressive extensibility: no built-in Model Context Protocol (MCP) support, no native sub-agents, no plan mode, no permission popups. Instead, these are all buildable as extensions via TypeScript modules, skills (markdown-based Agent Skills standard), and pi packages (shared npm/git modules). This design mirrors Unix philosophy—do one thing well and compose. The codebase is approximately 5,000 lines of TypeScript across core packages.

## 2. Agent Loop Architecture

The heart of Pi is the agent loop implementation in `packages/agent/src/agent-loop.ts` (743 lines). The loop uses a nested while-loop structure that handles both multi-turn conversations and queued follow-up messages:

```typescript
while (agentState.followUpMessages.length > 0 || !agentState.terminated) {
  // Outer loop: handles follow-up messages after conversation ends
  while (steeringMessages.length > 0 || !turnEnded) {
    // Inner loop: handles steering (user input during agent work) and tool calls
    const response = await lmFunc(...);  // Call LLM
    // Process response (tool calls vs. text)
    // Execute tools if present
    // Stream response, emit events
  }
}
```

The loop is driven by `runAgentLoop()`, which receives an `AgentLoopInput` containing:
- `lmFunc`: The LLM provider function (from pi-ai abstraction)
- `tools`: Available tools with schemas and handlers
- `systemPrompt`: System instructions and context
- `agentState`: Mutable state tracking conversation, tools, etc.
- Event emission callbacks (`streamFn`, event handlers)
- Hooks: `transformContext()`, `beforeToolCall()`, `afterToolCall()`, `shouldStopAfterTurn()`

**Key design pattern**: The loop is entirely synchronous in structure (using async/await internally) but is event-driven. State changes are emitted as events (`agent_start`, `turn_start`, `message_start`, `tool_execution_start`, etc.) that can be observed by the harness or UI layer. This decoupling allows streaming response display, progress indication, and even aborting mid-turn.

## 3. Tool Dispatch

Tools are defined with metadata including an optional `executionMode`. The `executeToolCalls()` function supports two execution modes:

**Parallel execution** (default): Tools run concurrently via `Promise.all()` over prepared tool invocations. This is the default for independent tools like file operations. The loop maintains result order to match the original tool call sequence.

**Sequential execution**: For tools marked with `executionMode: "sequential"` (e.g., bash commands that may have dependencies), the loop uses a for-loop with await. This ensures tool N+1 waits for tool N to complete.

Tool execution involves three phases:

1. **Preparation** (`beforeToolCall` hook): Validate arguments, check permissions, prepare execution context. If this hook returns an error, the tool is not executed and an error message is returned to the LLM instead. This is how the Pi harness implements permission gates—not as separate layers but as synchronous hooks that can reject tool calls.

2. **Execution**: Call the tool handler with validated arguments. If execution throws an error, it's caught and formatted as a tool error result (not a fatal agent error).

3. **Finalization** (`afterToolCall` hook): Post-process results (redaction, augmentation, logging). The hook can modify results before they're sent back to the LLM, enabling result transformation pipelines.

Built-in tools (`read`, `write`, `edit`, `bash`, `grep`, `find`, `ls`) are implemented in `packages/coding-agent/src/tools/` and registered into the toolsById map at harness initialization. Tool schemas are JSON Schema compatible.

## 4. Context & Conversation Management

**Compaction strategy**: When a session grows too large (configurable threshold, default ~100K tokens), the harness triggers compaction. Instead of truncating history (lossy), compaction runs a summarization pass via the LLM, producing a summary message that replaces the oldest N messages while keeping the full JSONL in file (for recovery). This preserves context for branching while keeping active context sizes manageable.

**Steering and follow-up messages**: The loop supports two types of queued user input:
- **Steering messages**: User input delivered during agent execution (before the turn ends). These are checked after tool execution and processed in the next iteration of the inner loop, allowing interrupt-like behavior.
- **Follow-up messages**: User input queued after the agent terminates. These are processed in the next iteration of the outer loop, automatically continuing the conversation.

## 5. System Prompts & Instructions

The agent's behavior is shaped by system prompts loaded from multiple sources and merged hierarchically:

1. **Global system prompt** (hardcoded or configurable at harness init): Provides baseline agent instructions
2. **Directory-specific instructions**: `CLAUDE.md` or `AGENTS.md` loaded from the current working directory and any parent directories. This allows different projects to customize agent behavior without code changes.
3. **Dynamic prompts**: Optional `systemPrompt` argument to `agentLoop()` overrides the above for a single conversation.

The coding-agent CLI implements a sophisticated resource management system:
- Skills are loaded from `~/.pi/agent/skills/` (markdown files following the Agent Skills standard)
- Extensions are TypeScript modules loaded at runtime and given access to the harness API
- Prompt templates are markdown files with variable substitution for reusable conversation starters

## 6. Permissions & Sandboxing

Pi does not have a dedicated permission layer. Instead, permissions are implemented via hooks in the tool execution pipeline:

**`beforeToolCall` hook**: This synchronous hook is called before every tool execution and receives the tool call with arguments. The hook can:
- Validate arguments against a policy
- Reject the tool call (returning an error)
- Modify arguments (e.g., sandboxing file paths)
- Require user confirmation (via a promise that resolves after user interaction)

This design pushes permission logic into the application layer rather than the framework. A harness can implement soft restrictions, hard rejections, user confirmation popups, or audit logging — all via hooks.

## 7. Sub-Agents & Delegation

Pi core does **not** support sub-agents natively. This is intentional. Instead, sub-agent behavior is built via:

1. **Tool-level delegation**: A tool can invoke another agent loop internally
2. **Extensions**: An extension can register a tool that dispatches to a sub-agent
3. **Prompt-based delegation**: Steering the LLM to break problems into subtasks within a single agent instance

## 8. Streaming & Parallelism

Streaming is a first-class concern. The `streamFn` parameter to `runAgentLoop()` is called with each token chunk from the LLM provider. The default `streamSimple` function writes chunks to stdout. The TUI variant (`streamTui`) buffers chunks and updates terminal display incrementally.

**Parallelism**: Tool execution is parallelized by default via `Promise.all()`. The loop collects all tool calls from the LLM response, prepares them concurrently, executes them in parallel, and collects results in order.

## 9. State Persistence

Sessions are persisted as JSONL files in `~/.pi/agent/sessions/{sessionId}.jsonl`. Each line is a JSON record representing a single message or metadata event. The session structure supports branching: each line includes an `id` and optional `parentId`, allowing users to fork sessions and explore alternative conversation paths (via `/fork`, `/clone` commands in the CLI).

**Compaction**: When a session's context grows large, compaction is triggered:
1. Summarize the oldest N messages via the LLM
2. Write a special "compaction" record to the JSONL (preserving the original messages in the file for recovery)
3. Load only the summary + recent messages into active context

This design preserves full auditability (the JSONL file is append-only and complete) while keeping active context manageable.

## 10. LLM Provider Abstraction

The `packages/ai` layer provides a unified interface to multiple LLM providers. Each provider (Anthropic, OpenAI, Google, etc.) implements this interface, normalizing tool schema formats, token streaming, error handling, and retries.

**Provider-specific optimizations**: The Anthropic provider leverages prompt caching (via the `cache_control` field on system and conversation messages) to reduce latency and cost on repeated contexts.

## 11. Standout / Unusual Design Choices

**Minimalist core with maximal extensibility**: Pi's core is deliberately small (~2,000 LOC for the loop and state machine). Everything beyond the bare loop—MCP, sub-agents, plan mode, permission popups—is buildable as an extension. This inverts the typical framework approach (batteries included) and enables users to build exactly what they need without bloat.

**Event-driven architecture**: Every meaningful state change in the loop emits an event. This decouples the loop from the UI, harness, and extensions—any observer can react to events independently.

**Hooks as the primary extension point**: Rather than plugin classes or middleware, Pi uses simple function hooks (`beforeToolCall`, `afterToolCall`, `transformContext`). These are stateless and synchronous, making them easy to reason about and compose.

**JSONL for persistence**: Using line-delimited JSON (instead of a database or binary format) for session storage is unusual but pragmatic. It's human-readable, append-only (safe for concurrent writes), and version-control-friendly. Branching is implicit in the file structure.

## 12. Implications & Design Patterns

**Composability over monoliths**: The four-package layering means each package is independently useful. You can use pi-ai to call LLMs without an agent loop, or use pi-agent-core in a headless application without the TUI.

**Permissions as policy, not gates**: By pushing permission decisions into hooks, Pi assumes the application (or extensions) will implement policy. This is more flexible than a built-in permission model but places responsibility on the user. The default behavior is permissive (all tools available), requiring explicit extension to sandbox.

**LLM provider neutrality but not agnosticism**: Pi supports many providers but doesn't hide their differences. The application must choose a model and provider upfront; the abstraction normalizes APIs but not capabilities.
